The one decoupling: CodeCharterBackend

Both worlds satisfy one four-method interface; the UI depends only on the interface and never produces data nor knows where it runs.

create_backend() in backend_provider.ts picks VSCode when acquireVsCodeApi exists, else MOCK. The host implements the interface over postMessage; MockBackend implements it in-process — render pipeline identical.

@code-charter/ui render pipeline

Three pure stages turn plain serializable rows into a laid-out diagram; the UI imports no SQLite, graphology, or Ariadne runtime.

1
render_flow → RenderedRows. NodeRow/EdgeRow from @code-charter/types; the UI's only data source. Orchestrated by code_chart_area.tsx.
2
Adapter. Pure boundary to React Flow shapes. custom_graph_to_react_flow.ts.
3
Layout. Compound ELK + LRU cache (positions excluded from key — they are outputs). graph_layout.ts · layout_cache.ts.
4
ReactFlow. @xyflow/react render surface. code_chart_area/code_chart_area.tsx · app.tsx.

Everything drawn arrives as data through the backend; positions are excluded from the layout cache key because they are outputs. render_flow is rebound each render, so the effect depends on the id, not the bound method.

Theming & position persistence

One theme source feeds the whole render surface; layout state persists per entry_point in a single localStorage slot.

Wire format A CallGraph is built from Maps, which do not survive JSON.stringify — it always crosses postMessage via serialize_call_graph/deserialize_call_graph (packages/types/src/call_graph_serialization.ts). RenderedRows and FlowSummary are plain data and cross untouched.

Per-flow persistence is out of scope for v1 — opening another flow overwrites the single slot.

@code-charter/vscode — the extension host

The host registers a command, opens a webview, and brokers every message between it and the core flow model.

Watchers exist but the webview is not pushed FileSystemWatcher + a 500ms debounce + an on_call_graph_changed emitter exist, but extension.ts never subscribes. Flow data is a per-request snapshot; a code edit shows on the next request, not pushed. Live re-sync is the Stop-hook path (see drift-sync.html).

Repo-relative paths are the same portable id space the store and drift engine use, so hydrated flow ids match the live graph. The webview is built by get_webview_content loading standalone.global.js + standalone.css; UIDevWatcher hot-reloads in dev mode.

Skeleton + hydrated flow merge

list_flows serves and merges two altitudes; render_flow dispatches on the id's origin.

graph.db is opened via open_graph_store and closed in finally; a missing file makes read_store_rows return null, so the surface is pure skeleton. Both hashes truncate sha256 to 16 hex.

Webview ↔ host ↔ Ariadne data flow

One round-trip per backend call; the webview never touches Ariadne or the store — it asks the host.

The honest seam Everything authored — labels, pins, layout, hand descriptions — lives in graph.db, written only by the drift path; the host is a pure read-and-project broker serving a snapshot. That is why the same diagram survives a code edit — the presentation layer reads preserved state, it does not regenerate it.

UI-side: editor_navigation.navigate_to_file posts navigate_to_doc in-context, else opens a vscode://file URL. Package graph: types → core → { drift, ui, vscode }.
