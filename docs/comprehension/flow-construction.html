Flow Construction

A flow is one functionality — usually one diagram. But Ariadne reads code syntactically, so when a call hides behind an indirection it splits one functionality across several diagrams. After every code-editing turn, an agent stitches the pieces back together.

problem syntactic blind spot repair after every turn who judges the agent who writes the scripts

The split: one functionality, two diagrams

Ariadne resolves calls by reading the source. const fn = route(key); return fn(); calls a variable — the real target was looked up in a registry at runtime. Ariadne sees no edge, so the real target floats free and becomes its own diagram.

function (call-graph node) resolved call edge the call Ariadne can't follow

Two diagrams for one feature. dispatch really runs handle_request, but only at runtime — through route(key)'s table lookup. The edge never exists in the source, so the reader sees two unrelated fragments. fixture: handler.ts · registry.ts · router.ts

The exact lines, so the gap is concrete

handler.tsthe unresolved call lives on line 5
1import { route } from "./registry";2 3export function dispatch(key: string): number {4  const fn = route(key);  // fn = a value pulled from a registry table5  return fn();  ◂ calls a variable — Ariadne resolves it to no in-graph function6}

This is the whole problem, in one line. The inventory in Phase 1 reports this exact site: { file: "handler.ts", line: 5, source_line: "return fn();" }.

The repair loop: who does what, when

After every turn that edits code, a sub-agent runs in its own context to put the pieces back together. Read top to bottom: the agent judges, the scripts read and write, the store holds the diagrams. The agent never touches the store directly.

a request (agent or script acts) a result returned judgement loops live only in the agent lane

The split between judgement and writing is the whole design: the scripts do the deterministic reads and writes, the agent does the judging. On a host with no agent, the list pass alone still produces correct (if fragmented) diagrams. SKILL.md · drift-reconciler.md

PHASE 1List the fragments, then decide which are one

The list pass hands the agent an inventory: every entrypoint in the changed files, each flagged is_orphan and carrying its unresolved call sites. The agent reads each site, greps for the indirection, and reaches one decision per group.

next step "no" branch "yes" branch

The agent weights its attention toward orphans — a doc-linked entrypoint is usually a genuine flow root. A bridge names the unresolved site by file and line, copied straight from the inventory. SKILL.md Phase 1 · agentic_modes.ts build_entrypoint_inventory

What --apply-stitch builds: one joined diagram

The same little graph, now whole. The two seeds merge into one flow; a dashed bridge arc spans the gap that Ariadne missed.

member of the flow resolved edge (Ariadne) agent's bridge over the missed call

The id is always the alphabetically-first seed, never chosen by the agent — identity stays deterministic so the diagram keeps its place across runs. The agent picks the grouping and the label; the engine picks the id. agentic_modes.ts apply_stitch · reconcile_stitch.test.ts

The evidence bar: a bridge must point at a real missed call

The agent can be wrong. So a bridge only persists if the named file:line actually holds an unresolved call in the live graph. The script checks; an uncorroborated bridge is silently dropped — but the umbrella still merges.

corroborated uncorroborated · dropped

This is what stops the agent from hallucinating connections. The grouping is the agent's call, but every edge it draws must point at a missed call the graph can confirm. agentic_modes.ts unresolved_call_span · reconcile_stitch.test.ts "rejects a bridge…"

PHASE 2Describe each member, once

The joined flow comes back listing every member. The agent writes one short sentence each and submits them. Re-submitting the same sentence costs nothing; changing it lands a revision.

Re-describing an unchanged repo is free, and correcting a stale sentence always lands — so the loop can run after every turn without burning tokens on work it already did. SKILL.md Phase 2 · agentic_modes.ts apply_descriptions

If anything fails, the diagram is still correct

The deterministic pass is the floor. Every way the agent can go wrong — or be absent — falls back to a correct, if fragmented, set of diagrams.

Four guarantees Empty inventory → both judgement phases short-circuit; the deterministic output already stands.
Agent crashes mid-loop → the store holds the singleton flows the list pass already wrote; nothing is half-written.
No agent on this host → the list pass alone produces one flow per entrypoint with docstring-or-placeholder descriptions.
Identity is deterministic → a flow's id is always its alphabetically-first seed, never the agent's choice, so diagrams keep their place.

The worst case is a fragmented-but-honest map, never a wrong one. The agent only ever upgrades what the scripts already guarantee. reconcile_stitch.test.ts: byte-identical to the default reconcile · a no-op leaves orphans as singletons

Engine reference

The deterministic engine the loop sits on. Each item is a pointer; the full picture lives in the sibling pages.

The skeleton

build_skeleton_flows(graph) turns the call graph into one flow per top-level entrypoint plus an unattributed bucket — what the diagram shows on a cold repo before any agent runs. See Core Engine.

Hydration

A skeleton stub is upgraded in place to a persisted agentic.flow at the same id. induce_members re-derives membership from the seeds on demand — never a stored leaf set, so a shared leaf belongs to many flows. See Core Engine.

Re-sync & retire

An existing flow re-extracts in place through one funnel, re_extract(file_set, origin='code-change'), re-anchoring relocated symbols. A superseded flow soft-deletes. Both ride the list pass. See Drift Sync.

The three bin modes

--list-entrypoints (reconcile + inventory), --apply-stitch (merge + retire), --apply-descriptions (persist text). A default mode runs the deterministic reconcile alone for agent-less hosts. See Drift Sync.
