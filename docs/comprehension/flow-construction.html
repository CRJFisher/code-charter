Flow Construction

A flow is the unit of comprehension — a functionality umbrella over one or more call graphs. It is constructed in two tiers: a deterministic skeleton that is always available, and a hydration that enriches it; a drift-reconciler sub-agent then judges which fragmented entrypoints are one functionality and what each member does.

Where the agent runs

The drift-reconciler is Claude — a bounded sub-agent (tools: Skill, Bash, Read, Grep) that runs in its own context so reconciliation does not flood the main session. It invokes the drift-sync skill, which orchestrates it through two judgement phases over three thin drift-reconcile bin modes. The --list-entrypoints mode runs the full deterministic reconcile and emits the changed neighbourhood's entrypoint inventory; the agent Reads and Greps the unresolved call sites, decides which entrypoints are one functionality, and applies its judgement through --apply-stitch and --apply-descriptions. The deterministic list pass stands on its own — an agent-less host still gets one flow per entrypoint with docstring-or-placeholder descriptions.

The judgement lives in the agent; every bin mode is a dumb read or a scoped write. The agent never writes the store directly — every mutation goes through the skill's bundled script. assets/agents/drift-reconciler.md · drift/reconcile/agentic_modes.ts · bin/drift_reconcile.ts

Two tiers of one flow

The deterministic skeleton is recomputed each session from the call graph and costs nothing to regenerate; the agentic hydration is a persisted upgrade written once and preserved thereafter. The line between them is the cost-of-regeneration seam.

The skeleton is the permanent cheap substrate; hydration upgrades a skeleton stub in place to the same id, so the agentic layer enriches rather than replaces. core/model/flow.ts

The deterministic skeleton

build_skeleton_flows(graph) is the path the VS Code extension renders first, on a cold repo with no store. Each top-level entrypoint seeds one flow; forward-reachability fills its interior; the dominant seed's symbol_path names it; everything reachable from no entrypoint falls into the single unattributed bucket.

1
Seeds. build_skeleton_flows sorts graph.entry_points by id and walks them in that order, so output is byte-stable across runs. One SkeletonFlow per entrypoint. build_skeleton_flows
2
Reachable. reachable_from(seed, graph) forward-traverses resolved call edges, visiting children in sorted id order; cycles guard, external/unresolved targets skip. The reachable set is the flow's member_count. reachable_from
3
Identity. flow_id_of(node) = the entrypoint's enclosing-free symbol_path (entrypoints are top-level, so the enclosing scope is []). The label is the entrypoint's name. flow_id_of
4
Rank + dedup. Sort by member_count descending, id ascending to break ties; de-duplicate by id keeping the first (largest) — two same-named methods on different classes collapse to one symbol_path.
5
Unattributed. Every node in no entrypoint's reachable set is gathered, sorted, into the single UNATTRIBUTED_FLOW_ID bucket, appended last — the catch-all for dead/unreachable code. UNATTRIBUTED_FLOW_LABEL

Membership is never a stored leaf set — induce_members re-derives it on demand, so a leaf reachable from two entrypoints legitimately belongs to both flows. core/model/flow.ts

Identity excludes the body A flow id is the dominant seed's symbol_path only — the content_hash is deliberately excluded. Editing an entrypoint's body never re-keys its flow; a rename moves the name segment, re-anchored by the resolver when the flow is persisted.

Hydration & induction

Hydration upgrades a skeleton stub into a persisted agentic.flow along a five-stage deterministic engine spine. Every stage runs headless: grouping is detect_code_umbrellas (one flow per entrypoint touching a changed file), bridges are deterministic heuristics, and descriptions follow the docstring-or-placeholder ladder. The drift-reconciler's judgement enters after this pass: the stitch post-pass (--apply-stitch) merges fragmented entrypoints into multi-seed umbrellas, and --apply-descriptions upgrades the placeholders.

deterministic engine spine (headless) drift-reconciler judgement (later post-passes) agentic-lane store write
1
Grouping — deterministic. detect_code_umbrellas walks build_skeleton_flows and keeps every non-unattributed flow whose induced members touch a changed code file — one flow per entrypoint, no judgement. The first MAX_FULL_CODE_HYDRATIONS get full (describe-bearing) hydration; the overflow lands as cheap singleton stubs, logged. drift/reconcile/reconcile.ts
2
Induce. induce_members re-derives the member set by forward reachability from the umbrella's seeds, plus any deterministic bridge endpoints — never a stored leaf set. core/model/flow.ts
3
Bridges = heuristics. build_bridge_edges and detect_meta_json_sub_agent_bridges infer cross-graph and skill sub-agent links as agentic.bridge edges at BRIDGE_CONFIDENCE_INFERRED = 0.5 (dashed render). A pure code flow writes bridges: []. core/agentic/bridge.ts · registry_detector.ts
4
Descriptions — deterministic. resolve_descriptions runs plan_descriptions and combines its buckets: docstrings verbatim, the symbol name as a placeholder for the rest. No model call; the flow hydrates complete. The agent's --apply-descriptions later overwrites the placeholders. drift/reconcile/describe.ts · core/agentic/describe_policy.ts
5
Persist. write_agentic_substrate + write_flow write the agentic.flow node and substrate row-by-row through scoped write_fields('agentic'), stamping last_synced_at, never rebuild_layer. drift/reconcile/hydrate.ts · flow_store.ts

Hydrate-then-merge: stage 1 always writes a correct fragmented state first, so an agent crash mid-judgement leaves valid singleton flows and agent-less hosts still get flows. Stitching and re-describing are additive refinements on top. drift/reconcile/reconcile.ts

The single most load-bearing write rule Persisting a flow must never call rebuild_layer('agentic') — that call is store-global and would delete every other flow's agentic content, defeating lazy per-flow hydration. Field writes go through write_fields(..., 'agentic'), so a user-promoted (renamed/pinned) flow node survives untouched.

Entrypoint stitching

Ariadne's call-graph extraction is syntactic: dynamic dispatch, registry lookups and callback wiring frequently fail to resolve, and each unresolved callee is promoted to its own top-level entrypoint — one functionality fragments into several singleton flows. Stitching is the drift-reconciler's first judgement phase, a post-pass over the deterministic hydration: List → Explore → Apply. The list pass reports the evidence; the agent Reads and Greps the indirection; --apply-stitch merges the fragments it judges to be one functionality.

1
List. --list-entrypoints runs the full deterministic reconcile, then build_entrypoint_inventory emits every entrypoint defined in a changed file, flagged is_orphan when no documentation edge links it. Each carries its unresolved_sites — enclosed calls none of whose resolutions land on an in-graph node — with the trimmed source_line as evidence. drift/reconcile/agentic_modes.ts
2
Explore. The drift-reconciler Reads the source_line at each unresolved file:line and the enclosing definition, then Greps for the indirection that hides the edge — registry register(...) calls, dispatch tables, re-exports, callback wiring — and decides which entrypoints are one functionality. The evidence bar: never stitch entrypoints whose connecting call site you have not read. assets/skills/drift-sync/SKILL.md
3
Apply. --apply-stitch stitch.json hydrates each umbrella as one multi-seed CodeUmbrella whose id is the alphabetically-first resolved seed's symbol_path (deterministic, never agent-supplied), with agentic.bridge edges at BRIDGE_CONFIDENCE_INFERRED = 0.5 stamped extractor_id: "agentic.stitch"; absorbed singleton flows retire. Unknown seeds, duplicate claims, and unresolvable bridge endpoints are skipped with a diagnostic. drift/reconcile/agentic_modes.ts · hydrate.ts

Idempotent: a re-run with identical input upserts the same rows and finds nothing left to retire. An unstitched orphan stays a singleton flow — which is correct, not a gap. A malformed contract exits 2; garbage content is skipped so partial progress beats none. drift/reconcile/agentic_modes.ts

Hydrate-then-merge — the deterministic floor stands first The list pass writes one singleton flow per entrypoint before any judgement. If the agent crashes mid-phase, the store is left in a correct fragmented state; an agent-less host stops here and still renders flows. Stitching only ever merges and retires on top of that floor — it never gates the deterministic output.
Bridge endpoints are symbol_paths, never raw SymbolIds Stitch bridge src_id/dst_id are flow-layer symbol_paths validated against build_symbol_path_index; persisted agentic.bridge edges live in the store's id space and survive re-parses. The id space is shared with deterministic bridges, so a stitch bridge and a heuristic bridge are indistinguishable to the renderer apart from extractor_id.

Skeleton ↔ hydrated reconciliation

The selector merges two populations: persisted hydrated flows take precedence, the deterministic skeleton fills the rest. Membership re-induces from rename-stable seeds. A superseded flow (dominant seed gone or renamed) is retired by the seed-gone soft-delete in reconcile.

Re-sync re-induces a flow from its stored seeds — the id is unchanged by construction; there is no identity remap. A flow whose dominant seed no longer resolves is retired by the seed-gone soft-delete in resync_persisted_flow (the orphaned row is harmless); a renamed seed re-hydrates as a fresh flow, and a freshly hydrated umbrella retires persisted flows whose seeds it subsumes via retire_flows_subsumed_by. drift/reconcile/reconcile.ts
