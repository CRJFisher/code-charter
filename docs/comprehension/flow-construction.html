Flow Construction

A flow is v1's unit of comprehension — an agent-detected functionality umbrella over one or more call graphs. It is constructed in two tiers: a deterministic skeleton that is always available, and an agentic hydration that enriches it.

raw / call graph (deterministic) agentic / hydrated pure engine logic store write unattributed / catch-all — grey = flow · clay = store write / bridge · dashed = inferred

Two tiers of one flow

The deterministic skeleton is recomputed each session from the call graph and costs nothing to regenerate; the agentic hydration is an expensive, non-deterministic upgrade persisted once and preserved thereafter. The line between them is the cost-of-regeneration seam.

The skeleton is the permanent cheap substrate; hydration upgrades a skeleton stub in place to the same id, so the agentic layer enriches rather than replaces. core/model/flow.ts

The deterministic skeleton

build_skeleton_flows(graph) is the path the VS Code extension renders first, on a cold repo with no store. Each top-level entrypoint seeds one flow; forward-reachability fills its interior; the dominant seed's symbol_path names it; everything reachable from no entrypoint falls into the single unattributed bucket.

1
Seeds. build_skeleton_flows sorts graph.entry_points by id and walks them in that order, so output is byte-stable across runs. One SkeletonFlow per entrypoint. build_skeleton_flows
2
Reachable. reachable_from(seed, graph) forward-traverses resolved call edges, visiting children in sorted id order; cycles guard, external/unresolved targets skip. The reachable set is the flow's member_count. reachable_from
3
Identity. flow_id_of(node) = the entrypoint's enclosing-free symbol_path (entrypoints are top-level, so the enclosing scope is []). The label is the entrypoint's name. flow_id_of
4
Rank + dedup. Sort by member_count descending, id ascending to break ties; de-duplicate by id keeping the first (largest) — two same-named methods on different classes collapse to one v1 symbol_path.
5
Unattributed. Every node in no entrypoint's reachable set is gathered, sorted, into the single UNATTRIBUTED_FLOW_ID bucket, appended last — the catch-all for dead/unreachable code. UNATTRIBUTED_FLOW_LABEL

Membership is never a stored leaf set — induce_members re-derives it on demand, so a leaf reachable from two entrypoints legitimately belongs to both flows. core/model/flow.ts

Identity excludes the body A flow id is the dominant seed's symbol_path only — the content_hash is deliberately excluded. Editing an entrypoint's body never re-keys its flow; a rename moves the name segment, re-anchored by the resolver when the flow is persisted.

Agentic hydration & induction

Does an agent search the repo and connect Ariadne's entrypoints into logical groupings? Only through the stitch seam. Hydration upgrades a skeleton stub into a persisted agentic.flow along a deterministic engine spine, and a model can speak in exactly two optional seams: the describe seam supplies member descriptions, and the stitch seam supplies boundary decisions — merging orphan entrypoints that syntactic analysis leaves split into one multi-seed umbrella. Both default to deterministic and headless: with no executor set, a flow's grouping is reachability, already computed by build_skeleton_flows — one entrypoint plus its reachable subgraph — and the engine induces members, infers bridges with deterministic heuristics, runs a deterministic-first describe ladder, and writes the result. "Agentic" names the storage lane the result is written to (lower-confidence, judgement-tier), not a runtime AI.

deterministic · default path describe seam (optional · model supplies text) stitch seam (optional · executor fills ConfirmedStitch[]) agentic-lane store write
1
Grouping — deterministic default, stitch-aware with an executor. detect_code_umbrellas runs build_skeleton_flows(graph) and keeps any flow whose induced members intersect a changed file — one entrypoint + its reachable subgraph. When ReconcileDeps.stitch_entrypoints is set, detect_and_stitch_code_umbrellas takes stage 1 instead: confirmed orphans merge into multi-seed umbrellas, unconfirmed/overflow orphans stay singletons, non-orphan flows take the deterministic path. drift/reconcile/reconcile.ts
1b
Stitch seam. build_candidate_stitches runs detect_gaps, filters orphan entrypoints in the changed neighbourhood, and pairs each (plus unresolved-shape orphans) against every other neighbourhood orphan as (source, target) candidates, capped at MAX_STITCH_CANDIDATES = 50. The executor returns ConfirmedStitch[] — merged_seeds + a justifying bridge span + an inference_rationale — union-found into multi-seed CodeUmbrellas whose id is the alphabetically-first seed's symbol_path; bridges and rationale flow through hydrate_code_flow unchanged. core/agentic/stitch.ts
2
Induce. induce_members re-derives the member set by forward reachability from the umbrella's seeds, plus any deterministic bridge endpoints — never a stored leaf set. core/model/flow.ts
3
Bridges = heuristics. build_bridge_edges and detect_meta_json_sub_agent_bridges infer cross-graph and skill sub-agent links as agentic.bridge edges at BRIDGE_CONFIDENCE_INFERRED = 0.5 (dashed render); stitch bridges join the same plumbing. A pure code flow writes bridges: []. core/agentic/bridge.ts · registry_detector.ts
4
Descriptions — the describe seam. plan_descriptions is deterministic-first: cache → docstring → optional batched LLM ≤ DEFAULT_DESCRIBE_CAP = 200 → name placeholder. A flow hydrates headlessly with no model call; this seam supplies text only. core/agentic/describe_policy.ts
5
Persist. write_agentic_substrate + write_flow write the agentic.flow node and substrate row-by-row through scoped write_fields('agentic'), stamping last_synced_at, never rebuild_layer. drift/reconcile/hydrate.ts · flow_store.ts

"Agentic" is the storage lane, not a runtime AI. Two model seams exist and both are optional: the describe seam supplies text, the stitch seam supplies boundary decisions. The drift-reconciler sub-agent fills both — it shells the bundled script into the headless drift-reconcile bin and, as the stitch executor, judges which orphan entrypoints belong together. core/agentic/stitch.ts · drift/reconcile/reconcile.ts

Does an agent group entrypoints? — Only through the stitch seam The deterministic default never does: an umbrella is a skeleton flow (one entrypoint + its reachable subgraph) and build_bridge_edges is a heuristic. When ReconcileDeps.stitch_entrypoints is set, the executor — the drift-reconciler sub-agent — judges candidate orphan pairs and its ConfirmedStitch[] merges them into multi-seed umbrellas, each stitch carrying a justifying bridge span and an inference_rationale. The describe seam supplies text; the stitch seam supplies boundaries; null_stitch_executor confirms nothing and keeps hydration byte-identical to the headless path.
The single most load-bearing write rule Persisting a flow must never call rebuild_layer('agentic') — that call is store-global and would delete every other flow's agentic content, defeating lazy per-flow hydration. Field writes go through write_fields(..., 'agentic'), so a user-promoted (renamed/pinned) flow node survives untouched.

Skeleton ↔ hydrated reconciliation

The selector merges two populations: persisted hydrated flows take precedence, and the deterministic skeleton fills the rest. Membership re-induces from rename-stable seeds; when a re-detection moves a flow's id, a ≥50% member overlap carries the user's label across the change.

Re-sync re-induces a flow from its stored seeds with the identity remap disabled — the id is unchanged by construction; only a genuine first-hydrate (a new id) runs the ≥0.5 overlap carry. drift/reconcile/hydrate.ts · flow_identity.ts
