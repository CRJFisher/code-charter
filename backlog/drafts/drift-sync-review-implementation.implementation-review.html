Code Charter · Drift Sync · Goal 1

Concurrency, Completeness, and the Missing Final Stage

Implementation Review 49-agent · Sonnet reads → Opus ×5 → Fable synthesis 8 improvements ranked

Drift sync is the machinery that keeps Code Charter’s flow diagrams honest: when you edit code, a background job re-reads the changed files and updates the stored graph the extension draws from. This review asks whether that machinery is correct. The engine that decides what changed is solid and well-tested. The trouble is upstream of it: three separate programs — the edit-triggered reconcile job, an occasional manual reconcile, and the VS Code extension — all read and write the same SQLite database file with no traffic control. No lock, no wait-your-turn setting, no atomic hand-off of the pending work list. So under any overlap a reconcile can silently die, staged edits can be deleted before they’re processed, and two runs can quietly overwrite each other. Every serious finding traces back to that missing coordination — and to a pipeline that never got its final “tell the UI” stage.

Improvements in detail

Each Improvement

#1 · CRITICAL SQLite concurrency: WAL + busy_timeout, read-only extension, single-reconcile lock

The SqliteGraphStore opens with no WAL and no busy_timeout. Three concurrent writers — Stop-hook subprocess, manual /drift, and the VS Code extension — share graph.db. Any contention throws SQLITE_BUSY: reconcile exits 1, pending set never consumed.

  • In SqliteGraphStore constructor (packages/core/src/storage/sqlite_graph_store.ts:38-42), set PRAGMA journal_mode=WAL and PRAGMA busy_timeout=5000 immediately after open
  • Open the extension’s store connection read-only (extension.ts read_store_rows never writes) so it never competes for the write lock; wrap the nodes+edges fetch in one read transaction for a consistent snapshot
  • Add a process-level reconcile mutex beside the store (O_EXCL wx lockfile in .code-charter/ or BEGIN IMMEDIATE advisory row); a second reconcile waits or exits 0 as a no-op since its files are already unioned into the pending set
packages/core/src/storage/sqlite_graph_store.ts  ·  lines 38–42  ·  CURRENT + FIX
constructor(db_path: string) {
  this.db = new DatabaseSync(db_path);
  this.db.exec("PRAGMA foreign_keys = ON");
  this.db.exec("PRAGMA journal_mode = WAL");
  this.db.exec("PRAGMA busy_timeout = 5000");
  this.initialize_schema();
}

#3 · HIGH Make the pending-reconcile handoff atomic

Two data-loss windows share one root. (a) Consume race: drift_sync.js unlinks the pending file after success — edits staged during the run are silently deleted. (b) Watermark divergence: the cursor advances on every Stop fire regardless of outcome; lost pending files are permanently skipped.

#5 · HIGH Persist a reconcile run log

Reconcile diagnostics — FlowOutcome, DeferredRetirement, hydration-cap notices — go to stderr in the Claude session, a different process, and are discarded. Nothing durable in graph.db answers “why was flow X retired?” or “which file set drove the sync?”

Implementation strengths

What the Engine Gets Right

The deterministic core is architecturally sound and unusually well-tested
Clean deterministic/agentic split with a graph-corroboration evidence bar: apply_stitch rejects invented bridges unless corroborated by an unresolved call site in the live graph — the agent can propose but not fabricate structure.
Scoped-upsert write model (flow_store.write_flow, re_extract) never does a store-global rebuild, preserving other flows’ agentic content; writes are idempotent and deterministic.
Careful within-process retirement discipline: deferred retirement when the graph is untrustworthy (empty graph, omitted seed file, zero-symbol parse), resurrection guards, and double-retire race defenses.
Rename-stable symbol_path anchor identity and a disciplined path-normalization funnel (paths.ts) across the hook → reconcile → store chain.
Unusually strong test coverage for deterministic paths: every hooks-layer and installer module has dedicated unit tests, and the delta/membership/code/stitch integration suites use real filesystems, real SQLite, and a real headless Ariadne pipeline.
The two-tier eval design (Tier 1 golden-wire structural + Tier 2 live stitch_eval with scaffolded fixture repos) is the right shape for measuring agentic judgement.
Error handling correctly prioritizes never breaking the host Claude session (hook exits 0 on any failure).
