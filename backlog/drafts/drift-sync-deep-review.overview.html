One Silent Root Cause Behind Two Review Goals

Overview · Family Landing 49 Agents · Sonnet → Opus ×5 → Fable Recipe: task-family-overview

Drift sync keeps Code Charter's flow diagrams in step with your code. Edit a file, end your turn, and a background job re-reads what changed, updates a graph of "flows" — clusters of related functions with plain-language descriptions — and the VS Code extension redraws. A 49-agent multi-model review asks two questions of that machinery. First: is it correct? The engine that decides what changed is sound and well-tested, but three programs share one database file with no coordination, and the pipeline never got its final "refresh the UI" stage. Second: is it pleasant to work on? No — inspecting a sync means reverse-engineering a SQLite schema by hand, and half the debug steps fail silently. Both answers trace to one root cause: the system stays quiet, so healthy no-ops and real bugs look identical. This page is the map; the two reviews are the territory.

The Three Structural Gaps

Gap 1 — Critical

Concurrency Vacuum

Three uncoordinated actors — the Stop-hook reconcile subprocess, an occasional manual /drift reconcile, and the VS Code extension — share graph.db and drift_pending_reconcile.json with no WAL, no busy_timeout, no lockfile, and non-atomic JSON writes. Any overlap causes a silent dropped reconcile (SQLITE_BUSY → exit 1) or a data-loss window (edits staged mid-reconcile deleted by the post-success unlink).

Gap 2 — High

Missing Final Pipeline Stage

The pipeline's stated contract ends in "flow store update → UI notification" but that link was never built. Nothing watches graph.db, so stitched flows and LLM descriptions landing there out-of-process are invisible until the developer manually re-runs Generate Diagram. The debounced-watcher pattern already exists in dev_watcher.ts — it just needs to be wired up.

Gap 3 — High (cross-cutting)

Silence by Design

The hook exits 0 on any error — stderr goes to the Claude session, not VS Code, and nothing records the outcome. No run log, no inspection tool: correctness bugs and healthy no-ops are indistinguishable.
