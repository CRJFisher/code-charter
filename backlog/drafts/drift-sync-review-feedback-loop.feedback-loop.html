Drift Sync · Feedback Loop Review

The Debug Loop Is the Binding Constraint

49-agent workflow Sonnet → Opus × 5 → Fable Goal 2 of 2 7 friction points 8 missing tools

When you change how drift sync works, you need to see what your change did. Right now you can’t — not easily. Today’s loop runs seven steps across three separate programs: rebuild the code (silently ignored if you forget), arm a test repo (fails invisibly if the build broke), start a whole Claude session to trigger a sync, then read the results in the Claude transcript, then open a SQLite command line and hand-write queries against a schema you have to reverse-engineer, then switch windows and manually redraw the diagram because nothing refreshes it. Three of those steps fail silently, and there is no “show me the before-and-after” anywhere. This review maps that loop, ranks its friction points, and proposes the missing tools — chiefly a store-inspection command and a single deterministic preview — that would shrink it to something you can run in seconds.

Friction Points — Detail

1
No First-Party Way to Inspect Sync Results
Impact

The core iteration question — is the sync working, and is it good? — is unanswerable without bespoke manual spelunking. Verified consequence: the live bergamot store's 0-bridge/1-member-edge anomaly sat unnoticed.

Solution

Ship a drift-inspect bin: summary mode (flow counts, per-flow members, bridge rationale, deferred retirements), --json, --flow <id> drill-down, and --lint (0-member flows, unpersisted stitch payloads, high placeholder ratio). Wire the same summary into the Code Charter OutputChannel.

2
Canonical Developer Docs Are Entirely Stale and Drift-Free
Impact

A developer following the docs is led entirely astray; the actual loop, including stitch_eval, is undiscoverable. A stale canonical doc is worse than none.

Solution

Rewrite both docs against reality: five packages, target-repo model (Dev Host → ~/workspace/bergamot), the install/reconcile chain, where graph.db lives, the full change→rebuild→trigger→inspect loop, the actual launch configs, stitch_eval as the quality loop, and the meaning of each .code-charter/ artifact.

3
Edit→Observe Loop Spans 3 Process Contexts, No Single-Command Path
Impact

Minutes-long, error-prone iteration for what should be a seconds-long deterministic loop; agent latency and token cost paid even for non-judgement changes; the hidden --dry-run capability goes unused.

Solution

Add npm run drift:dev -- --repo <path> --files <changed>: runs the deterministic reconcile against a scratch copy of the store and prints a before/after diff of flows/descriptions/bridges (no Claude session). Expose --dry-run as a documented drift:dryrun wrapper and a dev-mode "Preview Drift Reconcile" command printing would-be outcomes to the OutputChannel.

4
Every Intermediate Failure Is Silent
Impact

"Why did my sync do nothing?" is the most common debug question and currently has no starting point; the developer can iterate for a whole session against a disarmed or stale hook.

Solution

On activation/generate, verify the Stop hook in the target's .claude/settings.json and show a status-bar item ("drift armed" / "drift NOT installed — click to fix"); change the preLaunchTask to reveal-on-problem; persist last-attempt/last-success/last-error sync status beside the store and surface it in the OutputChannel; GC the 22 stale watermark files accumulated in bergamot.

5
No Webview Refresh After a Sync
Impact

Seeing a sync's visual effect requires manually re-running Generate Diagram. Makes it easy to misattribute a stale panel to a broken reconcile. Shared root cause with Goal 1 improvement #2.

Solution

FileSystemWatcher on graph.db → debounced webview auto-refresh. Same fix as Goal 1 improvement #2 — shared root cause, shared solution.

6
Quality Eval Covers Only Stitch Structure, at Cheapest Model, Undiscoverable
Impact

The only quality signal for the mechanism's actual value proposition (correct stitching + informative descriptions) has large blind spots; a prompt regression in SKILL.md/drift-reconciler.md ships undetected by CI.

Solution

Add a description-quality scoring pass (name-restatement heuristic or expected_description_contains goldens); add multi-umbrella and deep-chain fixtures; a CI guard on stitch_eval’s prompt-asset hashes; document the harness prominently.

7
CODE_CHARTER_DEV_MODE Does Nothing for the Drift Loop
Impact

Dev mode toggles only webview command URIs, the find widget, and the UI-bundle watcher — no store instrumentation, verbose logging, DB watching, or inspection affordance for the mechanism actually under development.

Solution

Extend dev mode: reveal the OutputChannel, watch graph.db and auto-refresh, print a store summary on each generate, expose "Dump Drift Store" and "Preview Drift Reconcile" commands.

Hidden Assets Already Built

Surface, don’t build
Much of the feedback-loop fix is surfacing what already exists, not building new machinery.

Four substantial capabilities exist in the codebase today but are unreachable, undiscoverable, or discarded. The path of least resistance is exposure — documentation, wiring, a wrapper script — not net-new construction.

dry_run_store / --dry-run Exists and works • unreachable & undocumented
stitch_eval.ts Best tooling in the system • undiscoverable from docs
Rich ReconcileResult diagnostics Produced correctly • discarded to ephemeral stderr
on_call_graph_changed Subscription hook on AriadneProjectManager • never wired
Verified consequence of friction point #1
A real anomaly sat unnoticed in the live store.

The bergamot store contains 34 flows and a stitch.json but zero agentic.bridge edges, only 1 flow_member edge, and 24 placeholder descriptions. The developer has no tool to notice it.

This is the human cost of friction point #1, made concrete. drift-inspect --lint would surface this class of anomaly on every run.
