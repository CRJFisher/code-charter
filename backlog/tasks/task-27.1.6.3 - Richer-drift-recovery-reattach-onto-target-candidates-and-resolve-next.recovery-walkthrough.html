Recovering a stranded description onto the symbol you actually meant

drift_list / drift_resolve MCP user-facing surface store-only · no new table parent task-27.1.6 (v1)
What this surface does

The auto-sync loop carries your authored descriptions and names across renames automatically. When it can't — the symbol is genuinely gone — the content lands in a recoverable re-attachment bin. Today the recovery tools handle two endpoints (commit a relocation; un-delete onto the original anchor) but not the case in the middle: the symbol is gone and you know the right new one. This task adds four capabilities to the same two tools so that case is reachable.

The three strandings — and where the gap is

When you stop working and code has moved, every preserved description gets re-resolved against the current code (the resolver cascade in resolve_anchor.ts). The verdict puts the description into one of three states. The recovery tools cover the two ends; this task fills the middle.

A · RELOCATED

The body moved, intact

The symbol was renamed or moved but its body is byte-identical somewhere new. The resolver finds it by content_hash and stages the new target on the node.

The node stays live, never binned. It carries drift_status:"relocated" staging attributes.

drift.resolve {reanchor} — commit the staged move. Authored fields ride across untouched; only the anchor changes.
B · GONE, NEW TARGET KNOWNTHE GAP

You know the right symbol

The body changed enough that no content_hash match exists — the resolver says miss, so the content is binned. But you know which current symbol it now describes.

Today the tools can't get you there: reattach only un-deletes onto the old, absent anchor, and the listing offers no targets to pick from.

drift.resolve {reattach, target} NEW — bind the binned content to a chosen current symbol.
C · GONE, NO TARGET

The thing is just gone

The code it described was deleted outright and there is no successor. The content is binned, preserved, never auto-pruned.

You either keep it parked, restore it onto its original anchor as a note, or delete it.

drift.resolve {reattach} (bare) restores onto the original anchor · {delete} keeps it removed.
The shape of the fix Case A and Case C already work. The whole task is reachability of Case B: give drift.resolve a target argument (capability 1), give drift.list ranked candidate targets so you don't have to hunt (capability 2), make node-vs-edge addressing explicit (capability 3, an integrity fix), and add a primitive to walk the bin one entry at a time (capability 4).

Reattach-onto-target: how authored fields thread across

The new path reuses the exact write that Case A already trusts — reanchor_node — which is the single place an anchor is rewritten while every authored field is preserved. The only new work is choosing the target from an argument instead of from staged drift attributes, and restoring the soft-deleted row first.

reattach-onto-target (new) bare reattach (still works) preserved authored fields

The load-bearing guarantee lives in reanchor_node: it writes only the structural anchor column and strips the drift_* staging keys; every authored field, including a user-owned description, is copied through verbatim. Reattach-onto-target gets edit preservation for free by routing through it.

packages/core/src/reextract/reanchor.ts:21–29
/** Strips drift-staging attrs, rewrites anchor, leaves everything else exactly as it was. */
export function reanchor_node(store: GraphStore, node: NodeRow, target: Anchor): void {
  const attributes = { ...node.attributes };
  const field_ownership = { ...node.field_ownership };
  for (const key of DRIFT_STAGING_KEYS) {
    delete attributes[key];
    delete field_ownership[key];
  }
  store.upsert_node({ ...node, anchor: format_anchor(target), attributes, field_ownership });
}

The current handler only ever calls this from reanchor, reading the target from staged attributes. Reattach-onto-target supplies the target from the argument instead, after un-deleting the row. The target must be a symbol that resolves today — so its (symbol_path, content_hash) comes from the same live index that produces candidates (next section).

packages/drift/src/mcp/drift_tool.ts — current reanchor path:88–97
function reanchor_resolve(store: GraphStore, id: string): DriftResolveResult {
  const observation = outstanding_drift(store).find((o) => o.node_id === id);
  const node = observation === undefined ? undefined : store.node(id);
  if (observation === undefined || node === undefined) return { /* applied:false */ };
  reanchor_node(store, node, { symbol_path: observation.to_symbol_path, content_hash: observation.to_content_hash });
  return { id, resolution: "reanchor", target_kind: "node", applied: true, reanchored_to: observation.to_symbol_path };
}
Why this is safe Reattach-onto-target is not a new write primitive. It is store.restore() + the already-trusted reanchor_node, differing from Case A only in where the target comes from (an argument resolved against the live index, not staged drift attributes). The edit-preservation invariant is enforced in one place for all three resolutions.

Candidate ranking — store-only, never a re-extraction

A chooser shouldn't have to hunt for the right symbol. Each drift.list entry will carry a ranked candidates[]. The hard invariant: candidates are computed from the store's live anchored nodes — never by asking Ariadne to re-extract from the MCP layer. The signals all come from data already in hand on each soft-deleted bin entry and each live node's anchor.

The three ranking signals

SignalSource (store-only)Strength
content_hash matchA live anchored node whose content_hash equals the binned entry's — the body survived somewhere. Equivalent to the resolver's relocated arm.strongest — same body
symbol_path similarityLive nodes whose symbol_path shares the file + chain + name stem with the stranded anchor (parsed via parse_anchor).strong — a likely rename
same fileAny live anchored node whose path equals the binned entry's path.weak — narrows the field

The bin entry already exposes its path and (via its row) its anchor; the live nodes expose theirs through store.all_nodes(). Building the candidate list is a join over those two — exactly what the resolver index already shapes for the cascade (by_symbol_path, by_content_hash), so the ranking can reuse the same index built from live nodes.

Invariant — no re-extraction from the MCP layer Candidates must be derived from the store's already-persisted live anchored nodes. The MCP surface stays user-facing-only: it never spins up Ariadne or re-parses files (that is the Stop-hook → sub-agent → drift-sync path's job). A bin that mass-empties under a pathological file (the task-27.1.6.2 fault) is exactly the kind of cross-coupling this boundary avoids.

The kind disambiguator — closing an integrity hole

Today drift.resolve takes only an id and recovers whether it's a node or an edge by looking it up in the bin (re_attachment_bin(store).find(c => c.id === args.id)). That rests on an unenforced assumption: that a node id can never equal an edge key. Nothing in the store contract guarantees this — GraphTarget is explicitly { kind: "node" | "edge"; id: string } precisely because the id alone is not self-describing.

!
drift_tool.ts:74, 79
The current recovery-by-find
const entry = re_attachment_bin(store).find(c => c.id === args.id) — then { kind: entry.kind, id: entry.id }. A collision would silently address the wrong target kind.

The fix is to let the caller pass kind: "node" | "edge" explicitly, so drift.resolve constructs the GraphTarget from the argument rather than reverse-engineering it. The store's soft_delete / restore already take a fully-qualified GraphTarget — the disambiguator simply stops the surface from guessing.

Resolve-next — walking the bin one entry at a time

The fourth capability is a loop primitive. Recovery is a sequence of single decisions, and a user or agent shouldn't re-read the whole bin between each. Resolve-next returns the next outstanding entry so the caller can drive a tight loop: list once → resolve → get the next → resolve → … until the bin is empty.

The data is already there: re_attachment_bin(store, scope?) returns every soft-deleted non-raw row, ordered. Resolve-next is a thin read — return the head of that list (optionally the head after the just-resolved id), so the primitive adds no new state and no new table.
