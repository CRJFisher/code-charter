Comprehension job 2 · task .1

The accept-dance, replaced by an inline re-anchor

A relocated symbol is one Ariadne can still find under a new symbol_path (a rename within the same file, or a move). Before .1, a relocation did not follow the rename automatically — it staged drift attributes onto the old node and waited for a human to accept the re-anchor through an MCP tool. With customisation agent-mediated everywhere, there is no human edit to protect, so the round trip was pure ceremony. It is gone.

The branch that was removed

The relocation branch lived in re_extract.ts (in packages/core — the core half of the strip, which is why .1 is “part 2”). It wrote staged-drift attributes instead of moving the anchor:

packages/core/src/reextract/re_extract.tsbefore → after
// BEFORE: stage a relocation as outstanding drift, wait for accept.
if (result.status === "downgrade" && result.reason === "relocated") {
  store.write_fields(target, {
    [DRIFT_STATUS_KEY]: DRIFT_STATUS_RELOCATED,   // staged, waits for accept
    [DRIFT_FROM_KEY]: anchor.symbol_path,
    [DRIFT_TO_SYMBOL_PATH_KEY]: result.state.symbol_path,
  }, "agentic");
}
// NOW: re-anchor inline — re-key the description side-node to the new path,
// update node.path + node.anchor, soft-delete the old-id row. Content rides
// across; no DRIFT_* keys, no staged finding, no accept gate.

Everything keyed off those staged attributes became dead and went in the same task: reanchor_node, outstanding_drift, the drift.resolve {reanchor} arm (the last arm), and with it build_drift_server, drift_tool, call_log, tool_names, the drift_mcp bin, the .mcp.json registration, the installer MCP wiring, the @modelcontextprotocol/sdk/zod deps, and the SessionStart hook (bin, banner module, installer spec, settings group, wire types). The relocated_targets describe-exclusion plumb through reconcile/hydrate was removed too — the re-keyed cache makes it redundant. resolve_db_path survived and lives in src/hooks/ (the Stop hook and the drift-sync skill resolve the store path through it).

The verdict subtlety — how a re-key avoids clobbering Verdicts are two-phase: computed against the pre-pass snapshot, then applied as all soft-deletes first, then all re-key upserts. A re-key whose target id already holds a live row (a hit, or a body-changed twin — the resolver prefers a path match over a relocation) skips the upsert rather than clobbering it, because that live row already tracks the same path itself.
Membership re-sync stayed intact — AC#4 A relocated member of a flow (not a flow seed) still re-syncs. That path is membership-drift detection via the symbol-level delta (task-27.1.6.4), independent of the staged-drift surface that was removed. A relocation fixture confirms the affected flow re-syncs with no regression: removing the accept-dance did not quietly drop membership re-sync.
The Stop-hook chain was kept The Stop hook stages a pending-reconcile file; that chain stays. Only the SessionStart hook and the drift MCP server were removed by .1.
Comprehension job 3 · task .3 · decision resolved

Demotion-by-wrapper: the case seed-gone retirement missed

Part 1 replaced the Jaccard overlap remap with seed-gone retirement: a persisted code flow is soft-deleted when its stored entry_points no longer resolve in the fresh graph. That covered two of the three cases the remap covered — rename and removal. It missed the third: demotion.

No data was lost (flow content is agent-regenerated), but before .3 duplicates accumulated. This was a genuine decision, resolved with the user. Three candidate mechanisms were on the table; the chosen one is the conjunction of the first two:

✓ A+BDemoted seed + subsumption

Chosen. Retire a flow when none of its stored seeds is still an entry point of the live graph (demoted) and its induced member set is subsumed by the just-written flow's — both judged against the live graph, evaluated on-demand after each flow is written.

The conjunction makes coexisting genuine multi-entrypoint flows that merely share members safe by construction — each still owns a live entry point, so it is never retired. On-demand: runs only off this turn's writes, against the in-memory persisted list, never a global sweep. A 3b loop guard skips a flow already retired by an earlier iteration's check, so it is not resurrected by its own later resync in the same pass. A stray superseded flow whose code is untouched lingers until its code next changes — the accepted trade-off of the on-demand model.
Shipped — d6a6a9f

BMember-set subsumption alone

Retire a flow when its induced member set is a subset of another live flow's — without the demoted-seed condition.

Catches demotion and other subsumption shapes in one rule. Alone, it could retire a deliberately-narrow flow whose seed is still a legitimate entry point. The conjunction with the demoted-seed test is exactly what fixes this.
Folded into the chosen rule as one half

CRecord coexistence

Decide duplicates are acceptable; document why and leave both flows live.

Zero code. The flow list would grow monotonically under refactoring — a comprehension product that accumulates stale flows undermines its own purpose. Would leave part 1's AC#2 claim formally false.
Rejected
The negative test that pins the safety AC#3 demanded that genuine multi-entrypoint flows which legitimately share members are not retired. The shipped test pins exactly this: two genuine entry points sharing members both stay live across a re-sync of the shared member. Subsumption alone never retires — the demoted-seed conjunction is what distinguishes “demoted, retire” from “still a legitimate entry, keep”. The wrapped-entrypoint scenario also runs end-to-end through the real Ariadne headless path.
Comprehension job 4 · task .4

Scoping retirement to the changed set, and guarding a degenerate graph

Seed-gone retirement is now scoped to the changed set: a seed-gone code flow is surfaced for retirement only when this turn's changed files include the seed's defining file (parsed from the stored entry_points symbol_paths). An unrelated edit — or a degenerate whole-repo graph — can never retire a flow whose code was not touched. Where a flow is implicated, retiring still requires trustworthy evidence.

The second half of .4 is observability. Before it, the retire branch returned undefined, so retirements never appeared in ReconcileResult.outcomes, the --json output, or the summary line. Now FlowAction carries a retire action, the retire branch emits an outcome record, deferrals ride ReconcileResult.deferred_retirements, and the CLI summary counts both:

packages/drift/src/reconcile/reconcile.tsretire is now a first-class outcome
// BEFORE: the retire branch returned undefined — invisible everywhere.
deps.store.soft_delete({ kind: "node", id: flow.node.id });
return undefined;   // ← invisible to outcomes, --json, summary
// NOW: emit a retire outcome, or defer with a reason.
if (graph_degenerate(graph, seed_file, adapter)) {
  result.deferred_retirements.push({ id: flow.node.id, reason });
  return undefined;   // deferred — retried next turn that touches the file
}
deps.store.soft_delete({ kind: "node", id: flow.node.id });
return { action: "retire", id: flow.node.id, reason: "seed entrypoint gone" };

The CLI summary now reads reconciled N flow(s) (R retired) over M file(s); deferred K retirement(s), and the drift-sync SKILL.md documents the hydrate/resync/retire vocabulary, the retire record's degenerate fields, and the linger-vs-defer distinction.

Why .4 landed before .3 The outcome surface and the guard were plumbed first, so that when .3's demotion check arrived it was observable and safe from birth — demotion retirements flow through the same FlowAction "retire" outcome. The two tasks share affected_flows.ts and reconcile.ts; building the safe surface before the new retirement source meant no window where retirements were silent.

The agent-mediated customisation model, in one picture

Every strip in this family rests on one principle that, before .6, was recorded only in the task docs — doc-5 and doc-5.1 still described the deleted preservation surface. Task .6 wrote the principle in: customisation is agent-mediated at every layer — what you want the diagram to say you express to the agent, which authors it and re-applies it on every sync; no diagram layer stores direct human byte-edits.

Recurring vocabulary, decoded

seed
The entry-point symbol whose symbol_path is a code flow's identity. A flow's members are induced from its seeds by graph traversal, never enumerated.
seed-gone retirement
Soft-deleting a code flow when its stored entry_points no longer resolve in the fresh graph. The surviving lifecycle path that replaced the Jaccard remap; now scoped to the changed set and guarded.
relocated
A symbol Ariadne still finds under a new symbol_path (rename/move). Distinct from a miss (gone entirely). Now re-anchored inline by re-keying the description side-node.
accept-dance
The staged-drift round trip: stage a relocation, announce it, wait for an explicit drift.resolve {reanchor}. Removed by .1.
demotion
An existing flow's seed wrapped by a new caller, so it stops being an entry point but still exists as an interior node. The gap .3 closed with demoted-seed + subsumption.
on-demand
Retirement checks run only off this turn's writes, against the in-memory persisted list, never as a global sweep over untouched flows.
agent-mediated
Customisation the agent authors and re-applies each sync, rather than human byte-edits stored in a protected field. The invariant .6 records.
