Comprehension job 2 · task .1

The accept-dance, and the inline re-anchor that replaces it

A relocated symbol is one Ariadne can still find under a new symbol_path (a rename within the same file, or a move). Today, a relocation does not follow the rename automatically — it stages drift attributes onto the old node and waits for a human to accept the re-anchor through an MCP tool. With customisation now agent-mediated everywhere, there is no human edit to protect, so the round trip is pure ceremony.

The branch being removed

The relocation branch lives in re_extract.ts (in packages/core — this is the core half of the strip, which is why .1 is “part 2”). Today it writes staged-drift attributes instead of moving the anchor:

Comprehension job 3 · task .3 · decision needed

Demotion-by-wrapper: the case seed-gone retirement misses

Part 1 replaced the Jaccard overlap remap with seed-gone retirement: a persisted code flow is soft-deleted when its stored entry_points no longer resolve in the fresh graph. That covers two of the three cases the remap covered — rename and removal. It misses the third: demotion.

No data is lost (flow content is agent-regenerated), but duplicates accumulate, and part 1's AC#2 claim — “the seed-gone path covers the cases the remap covered” — is not fully true. This is a genuine decision. Three candidate mechanisms, with trade-offs:

ASeed-not-an-entrypoint

Retire when a flow's dominant seed is no longer an entry point of the live graph (not merely “no longer a node”).

Directly targets demotion: a wrapped seed stops being an entry point. Small change — narrow the resolution test from “node exists” to “is in graph.entry_points”. Risks retiring a legitimate multi-entrypoint flow whose seed is reachable from another entry but still validly an entry too — needs the AC#3 negative test. Entry-point set must be trustworthy — couples to .4's graph-health guard.
Likely pick — closest to intent

BMember-set subsumption

Retire a flow when its induced member set is a strict subset of another live flow's members.

Catches demotion and other subsumption shapes (merges) in one rule. Independent of entry-point classification. O(flows²) set comparisons per reconcile; the most expensive option. “Strict subset” is fragile — a one-member difference defeats it; near-subsets aren't caught. Could retire a deliberately-narrow flow the user wants kept distinct.
Heavier; broader but riskier

CRecord coexistence

Decide duplicates are acceptable; document why in this task and leave both flows live.

Zero code; honest if duplicates are rare and self-healing. Avoids the false-retirement risk of A and B entirely. The flow list grows monotonically under refactoring — a comprehension product that accumulates stale flows undermines its own purpose. Leaves part 1's AC#2 claim formally false.
Fallback only
What the AC requires either way AC#1 demands exactly one live flow after demotion (or a recorded coexistence decision). AC#3 demands a negative test: genuine multi-entrypoint flows that legitimately share members are not retired. That negative case is the sharp edge of options A and B — whichever mechanism is chosen must distinguish “demoted, retire” from “still a legitimate entry, keep”.
Comprehension job 4 · task .4

Where the degenerate-graph risk enters, and where the guard sits

Seed-gone retirement runs globally on every reconcile — against the whole freshly-built call graph, regardless of which files the turn touched. The judgement is only as good as that graph, and the graph can be silently wrong.

The second half of .4 is observability. Today resync_persisted_flow returns undefined on the retire branch (reconcile.ts:146-155), so retirements never appear in ReconcileResult.outcomes, the --json output, or the “reconciled N flow(s)” summary. The skill consuming the result cannot see or report that flows were retired.

  }
  ...
}
Why .4 comes after .3 The new outcome surface should carry both seed-gone and demotion retirements, and the guard should wrap whatever final retirement shape .3 lands on. Build .3's mechanism first, then make it safe and observable in .4. The two tasks share affected_flows.ts and reconcile.ts.

Recurring vocabulary, decoded

seed
The entry-point symbol whose symbol_path is a code flow's identity. A flow's members are induced from its seeds by graph traversal, never enumerated.
seed-gone retirement
Soft-deleting a code flow when its stored entry_points no longer resolve in the fresh graph. The surviving lifecycle path that replaced the Jaccard remap.
relocated
A symbol Ariadne still finds under a new symbol_path (rename/move). Distinct from a miss (gone entirely).
accept-dance
The staged-drift round trip: stage a relocation, announce it, wait for an explicit drift.resolve {reanchor}. Removed by .1.
demotion
An existing flow's seed wrapped by a new caller, so it stops being an entry point but still exists as an interior node. The gap .3 closes.
agent-mediated
Customisation the agent authors and re-applies each sync, rather than human byte-edits stored in a protected field. The invariant .6 records.
