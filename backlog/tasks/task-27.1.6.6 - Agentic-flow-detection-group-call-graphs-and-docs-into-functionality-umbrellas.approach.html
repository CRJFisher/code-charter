task-27.1.6.6 · drift · agentic entrypoint stitching · approach review

Reconnect the call-graphs Ariadne's syntactic analysis leaves split

Ariadne resolves call references structurally. Dynamic dispatch, registry lookups, re-exports, and callback wiring frequently fail to resolve — and every unresolved call site is a real caller→callee edge missing from the graph. A callee only ever reached through such a call has no in-graph caller, so it is promoted to its own top-level entrypoint, and one functionality fragments into several flows. This task adds the agentic pass that stitches those fragments back together by repairing the missing edge — through a single new injected seam, reusing almost everything the shipped engine already does. This page lays out that approach so it can be approved before any code is written.

Why this matters

Ariadne is a syntactic call-graph extractor: it resolves a call by matching it structurally to a definition. When a call goes through a registry table, a dynamic dispatch, a re-export, or a callback handed off elsewhere, the match fails — the call site resolves to nothing. Each such miss is a real caller→callee edge that never enters the graph. The downstream artifact is concrete: a callee that is only reached through unresolved calls looks like it has no caller, so build_skeleton_flows promotes it to its own top-level entrypoint. One functionality therefore fragments into several flows — one per spuriously-promoted entrypoint — and the v1 deterministic detector (detect_code_umbrellas) faithfully emits each as a separate single-seed flow, because it cannot tell a real entrypoint from a resolution gap.

The picture, at its smallest

Entrypoint A contains a call site that really targets entrypoint B, but the call is dynamic dispatch and does not resolve. Today: B has no in-graph caller, so A and B surface as two flows. Stitched: the agent confirms the missing edge A→B; A and B merge into one multi-seed flow, and the recovered edge is recorded as an agentic.bridge whose click-through lands on the real missed call site.

The deterministic substrate (task-27.1.4) already surfaces the raw material for the repair, but has no runtime caller at all — this task is its first consumer. find_unresolved_shapes flags the nodes whose call sites mostly fail to resolve (the registry / dynamic-dispatch shapes); derive_candidate_seeds enumerates the orphan entrypoints. From these the engine builds candidate stitches: each pairs an unresolved call site in one entrypoint's tree with the entrypoint(s) it might actually target.

The one architectural move

Add an optional executor stitch_entrypoints?: EntrypointStitchExecutor to ReconcileDeps, mirroring the existing describe?: DescribeBatchExecutor exactly. Undefined ⇒ today's deterministic one-entrypoint-per-flow behaviour, byte-identical. Supplied ⇒ the stitch lane: build candidate stitches from find_unresolved_shapes + derive_candidate_seeds over the changed neighbourhood, hand them to the executor, and assemble each confirmed stitch into a multi-seed CodeUmbrella plus an agentic.bridge over the recovered edge. Everything downstream is the existing path. The judgement asked of the agent is narrow and checkable: does this specific unresolved call really link these two entrypoints? — answerable from the two definitions and the call site, not "what is the essence of this repo?".

★How, when, and where the agent enters

The reorientation is precise about when the model is in the loop and when it is not. The seam is filled the same way the shipped describe seam is filled — the drift-reconciler sub-agent's own run — with no new harness and no background spawn. With nothing injected, the path is byte-identical to today and makes no model call.

Where

An injected executor seam on ReconcileDeps: stitch_entrypoints?: EntrypointStitchExecutor, added exactly like the shipped describe?: DescribeBatchExecutor.

Its default is a deterministic no-stitch grouper that reproduces today's one-entrypoint-per-flow. No executor ⇒ opt-out is the headless default; stitching is opt-in.

When

On the Stop-hook reconcile of the files worked on this turn, inside HYDRATE / re-detection, after re_extract refreshes the call graph and the substrate proposes candidate stitches.

Only the changed neighbourhood's entrypoints are considered — lazy and per-flow, never the whole repo.

How

Filled by the drift-reconciler sub-agent's own run — the same place the describe executor is filled. No new harness, no background spawn (task-27.1.5's separate execution model is superseded — YAGNI).

The agent receives the candidate stitches, each entrypoint's neighbourhood summary, and the goal, and returns the confirmed stitches. The in-process default makes no model call.

1The seam decision

The stitch is a judgement seam, not a new harness — the same pattern describe? already follows. With no executor injected, the headless path stays deterministic and the existing tests are untouched. The two paths differ only in how the set of CodeUmbrellas is produced; they share every downstream step.

The executor contract

input
{ goal, candidates: CandidateStitch[], neighbourhoods } — each CandidateStitch pairs an unresolved call site (from UnresolvedShape) with the entrypoint(s) it may target (from CandidateSeed), plus each entrypoint's neighbourhood summary. The goal threads through unchanged (AC#7), so a later goal selector is an added arg, not a rewrite.
output
ConfirmedStitch[] = { label, rationale, seeds (≥1 SymbolId), bridges: BridgeCandidate[] } — one entry per merged umbrella the agent confirms.
default
a pure no-stitch function returning one single-seed group per candidate entrypoint ⇒ reproduces today's one-entrypoint-per-flow exactly, no bridges.

3The stitch, on one example

Two entrypoints, runExport and flushQueue, are really one functionality: runExport dispatches to flushQueue through a handler registry, so Ariadne cannot resolve the call and promotes flushQueue to its own entrypoint. The agent confirms the missing edge; the two merge into one multi-seed umbrella, and the recovered call edge is recorded as an agentic.bridge with the call-site span as provenance — click-through lands on the real missed dispatch. The umbrella's id is its dominant seed — flow_id_of of the seed with the largest reachable subgraph, ties broken by path — so a single-seed (un-stitched) default derives the same id as today's skeleton flow (byte-identical), while a stitched umbrella gets a stable id that survives a later re-stitch.

// induce per-seed reachable size, then pick the dominant seed for the id.
const dominant = [...stitch.seeds]
  .map((s) => ({ s, size: reachable_from(s, graph).size, path: flow_id_of(graph.nodes.get(s)!) }))
  .sort((a, b) => b.size - a.size || (a.path < b.path ? -1 : 1))[0];

4Identity & preservation across re-stitch

A stitch is a judgement, so a later run can stitch or un-stitch differently. The guarantee: a re-run never silently drops a user rename or pin. Two existing mechanisms deliver it, unchanged. First, the id is the dominant seed's path, so an umbrella has a stable identity that survives a non-deterministic re-stitch as long as its dominant seed persists. Second, when a re-stitch does change the id, the ≥50% overlap remap (match_existing_flow, threshold 0.5) recognises the freshly-detected flow as the same flow under a new id, carries the user-owned label across (re-stamped user-owned so a later agentic pass cannot overwrite it), and strands the superseded flow — node + member edges — into the re-attachment bin.

carry user-owned content strand superseded flow → bin
AC#5 — already proven for single-seed; this task adds stitch/un-stitch fixtures

The shipped engine already runs this remap for single-seed flows. This task confirms it carries labels across stitch and un-stitch for multi-seed flows and adds fixtures for both directions. Nothing in the identity machinery is new code — the work is the fixtures and the confirmation.

Bounded cost & the proving test

Cost is bounded (AC#8). The stitch call is capped at a ceiling; entrypoints / candidate stitches beyond the cap — or stitches the agent declines — fall back to singleton stub flows (the un-stitched default), and any truncation is logged, never a silent cap. This mirrors the existing per-turn full-hydration cap (MAX_FULL_CODE_HYDRATIONS) already in the pipeline.

1

Cap the stitch call

The task-27.1.1 ceiling budgets the one stitch call. Candidate stitches above the cap are not sent to the agent.

2

Overflow / declined → singleton stubs

Over-cap or declined stitches fall back to describe-less singleton stub flows — exactly today's cheap default, so a repo never blocks on the agent.

3

Log the truncation

Any cap is written to the diagnostics sink (deps.log), so the reduction is visible, never silent.

AC#9 — the proving target (a registry / dynamic-dispatch fixture)

Like the describe seam, v1 wires no model-backed executor through the drift-reconcile bin — the stitch path is structural. The proving test injects a hand-written stitch executor on a fixture where Ariadne leaves two entrypoints split by an unresolved call site (a registry / dynamic-dispatch shape), and verifies that with the executor they collapse into a single multi-seed umbrella spanning both — an agentic.bridge over the missed call, a sensible label and rationale — versus today's two singleton flows; without an executor the two stay separate (byte-identical). The deterministic skill-dir flow (umbrella = directory) is unchanged.

Out of scope (deferred — YAGNI)

This task does not invent functionality groupings from nothing and does not attach docs by judgement. The broad "name arbitrary umbrellas / attach docs by judgement" framing is deferred; doc linkage stays deterministic (the skill-dir flow), and the depth/salience goal of key-control-flow is task-27.1.7.
