Why this exists

Task-27.1.6 turns the inert substrate the earlier subtasks built into a live loop. Before it, no production path populates the on-disk graph store and the drift-sync skill is a no-op stub — so a hydrated flow can never reach the webview. This task wires the full Stop-hook → sub-agent → skill → reconcile engine → store path end to end, doing double duty: HYDRATE a flow on first work, RE-SYNC it on later edits — always updating, never gating, always re-applying your authored description / name / pin.

The loop the parent ships

One Stop-hook entry point, two branches, with preservation woven through both. A genuine miss — content whose anchor no longer resolves, or a split/merge that strands a name — goes to the re-attachment bin, recoverable and never auto-pruned.

No review apparatus in v1

No observation/adjudication tables, no open→triaged→resolved lifecycle, no cosmetic/intent classifier, no PreCommit gate, no drift inbox. The diagram absorbs drift out-of-band, off your attention. Those are deferred to task-27.1.9 / task-27.1.10.

How the sub-tasks fit together

The first five children are the come-back-and-make-it-real pass on the live loop: prove and sand the recovery surface, fix a re-sync bug the loop surfaced, extend recovery, sharpen change detection so drift follows the code rather than the file, and trim the Stop-hook transport. The sixth opens the next capability — the agentic (L1) lane: it stitches back together the call-graphs Ariadne's syntactic resolution leaves split. Where a dynamic dispatch or registry lookup fails to resolve, one functionality fragments into several entrypoints; the agent repairs the missing edge so the pieces re-merge into one flow. The seventh makes that lane measurable in two tiers: committed mini-codebase fixtures parsed by real Ariadne, evaluated through the same three bin modes — a deterministic golden-JSON tier that runs in CI, and a live, env-gated tier that drives the real agent and scores its semantic judgment against authored ground truth.

Fine-grained change detection

Drive drift sync from the symbols that changed, not the files that were touched — a turn's edits reduce to an add/remove/modify/relocate SymbolDelta (from the resolver's existing content_hash anchors) that scopes which flows re-sync and which symbols re-describe. An edit to a shared file that touches no symbol a flow depends on no longer re-syncs it.

task-27.1.6.6 · Done

Agentic entrypoint stitching — reconnect the split call-graphs

Ariadne resolves calls syntactically, so dynamic dispatch, registry lookups, re-exports, and callback wiring frequently fail to resolve — and a callee reached only through such a missed call has no in-graph caller, so it is promoted to its own entrypoint. One functionality fragments into several flows. This adds the agentic pass that stitches them back together, with the intelligence owned by the drift-sync skill rather than any in-process function. The skill orchestrates the drift-reconciler sub-agent through two judgement phases over three thin store-facing bin modes. Phase 1 — stitch: --list-entrypoints emits the changed neighbourhood's orphan/unresolved inventory; the agent explores the codebase itself (grep/Read) to decide which orphans belong to one functionality, writes stitch.json (umbrellas = the existing CodeUmbrella shape), and --apply-stitch hydrates each multi-seed umbrella plus an agentic.bridge over the recovered edge. Phase 2 — describe: the agent reads each member and --apply-descriptions persists a short sentence apiece. Two whole TypeScript layers delete — the injected executor seams (stitch_entrypoints?/describe? + null defaults) and the candidate-pairing / union-find / agenda engine that fed them — because the agent, not the engine, now decides the flow shape. No orphans ⇒ both judgements skip and the deterministic resync/retire is byte-identical to today. Identity stays purely deterministic — id = dominant seed's symbol_path (flow_id_of); doc-attachment-by-judgement is deferred.

task-27.1.6.7 · To Do

Eval harness for the stitcher — mini-codebase fixtures

27.1.6.6's three bin modes are proven by spawning the built bin over one on-disk dispatch fixture; what is not yet measurable is the agent's semantic judgment — does it stitch only what is real? This task commits three mini TypeScript codebases (≤100 lines each) under __fixtures__/stitch_eval/ — registry_dispatch/ and callback_wiring/ (Ariadne genuinely cannot resolve their dispatch/callback calls, so they must stitch) and unrelated_pair/ (independent domains, must not stitch) — parsed by real Ariadne, and evaluates them in two tiers over the identical three modes production uses. Tier 1 (reconcile_stitch_eval.test.ts, jest, deterministic, in CI) feeds golden JSON through the modes: --list-entrypoints yields a well-formed inventory whose unresolved sites point at real lines; a golden stitch.json through --apply-stitch yields the correct multi-seed umbrella with a bridge over the real call site, and no umbrellas leaves singletons — guarding the structural / replay contract. Tier 2 (a runnable stitch_eval harness, env-gated, excluded from CI) drives the real drift-sync skill / drift-reconciler via claude -p per fixture, lets the agent itself run the three modes, then inspects the store and scores: two stitches, one principled decline, every member described — emitting a per-fixture report a SKILL.md edit re-runs. That live loop is the point of the task.
