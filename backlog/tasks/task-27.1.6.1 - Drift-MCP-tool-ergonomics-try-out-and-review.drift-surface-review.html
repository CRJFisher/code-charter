TL;DR — the decision you are confirming

KEEP the surface: the drift_list/drift_resolve naming, the id-only addressing, and the reattach/delete/reanchor vocabulary all hold up under a real run. REVISE exactly one thing here — enrich drift_list's payload with the stranded authored content so a user can tell what they are recovering. FOLLOW-UP the deeper limitation: reattach can only undelete onto the old, now-gone anchor — recovery onto a new symbol after a true miss is unsupported.

Two recovery paths — and which one is proven

The conceptually hardest part of this surface is that there are two distinct recovery paths that are easy to conflate. They diverge the moment a rename happens, on a single question: did the body change too?

relocated → reanchor (already proven) miss → bin → reattach/delete shared entry / decision
The sharpest gap — sits on the right path

drift_list's payload is too thin to choose with confidence (AC#2). A user on the miss path sees an opaque id + a timestamp and cannot tell what they are about to recover. The stranded authored text — the very thing they are trying to get back — is not in the entry.

What drift_list returns today

packages/drift/src/mcp/re_attachment_bin.ts — DriftBinEntry
export interface DriftBinEntry {
  kind: "node" | "edge";
  /** Node id or edge key. */
  id: string;
  layer: Layer;
  /** The node's file path; null for edges. */
  path: string | null;
  /** ISO-8601 soft-delete timestamp. */
  deleted_at: string;
  // ← no description / name: you cannot see WHAT you are recovering
}

The revise that lands here — enrich the payload (additive, AC#4-safe)

The bin query already reads the full node row (store.all_nodes({ include_deleted: true })), so the authored fields are in hand — they are simply dropped. Surfacing them is cheap, reads existing attributes, and needs no schema change and no new table, so it stays additive to task-27.0's reservations.

re_attachment_bin.ts — proposed enrichment (illustrative shape)
export interface DriftBinEntry {
  kind: "node" | "edge";
  id: string;
  layer: Layer;
  path: string | null;
  deleted_at: string;
  /** The stranded authored text, so the caller can choose with confidence. */
  description: string | null;
  name: string | null;
}
// populated from node.attributes already loaded by the bin query — no extra read.

The keep / revise / follow-up board

Each ergonomics dimension from AC#2, scored against a real run, with the AC#3 verdict.

Keep

Naming — drift_list / drift_resolve

AC#2 · AC#3

The underscore is forced, not a choice: a dot in an MCP tool name is rejected by clients that flatten the namespace (Claude Code surfaces a tool as mcp__<server>__<tool>, where <tool> must be [a-z0-9_]). The conceptual drift.* family reads cleanly; the server is named drift. No change.

Keep

Id-only addressing

AC#2

drift_resolve recovers the target's kind/space from the store, so the caller passes only an id. It works in the real run. It rests on an unenforced node-id vs edge-key disjointness — drift_resolve does a .find(c => c.id === args.id) over nodes-then-edges. Adequate today; a kind disambiguator is parked as a follow-up, not a blocker.

Keep

Resolution vocabulary — reattach / delete / reanchor

AC#2 · AC#3

The surface already grew past the task doc's premise. The doc (2026-06-02) describes a two-word reattach/delete vocabulary; the code now has three resolutions. reanchor commits a staged relocation onto the renamed symbol, carrying the user description across untouched — exactly the right affordance for the relocated case, and it is already wired into the SessionStart banner. The vocabulary is correct as-is.

Keep

Discoverability — banner → list → resolve

AC#2

The read-only SessionStart banner reports the outstanding-drift count and points the user at drift.resolve { reanchor } for relocations and drift.list for the bin. The loop a user/agent actually follows is intact. No change.

Revise

Payload richness — drift_list output

AC#2 · AC#3 · AC#4

The one change that lands here. Enrich DriftBinEntry with the stranded authored fields (description / name) so a user can tell what they are recovering. Cheap (the row is already loaded), additive (no schema change → AC#4 holds), and the highest-leverage ergonomics fix found.

Follow-up

Reattach-onto-a-new-target + candidate suggestions

deeper limitation

Named, not built here. reattach only undeletes onto the old, now-gone anchor — it does not re-point stranded content onto a new symbol. Only reanchor re-points, and only for the staged-relocation case. Genuine recovery-onto-a-new-symbol after a true miss is unsupported. Candidate follow-ups: a reattach-onto-new-target affordance + candidate-target suggestions in drift.list; a kind disambiguator on resolve; a "resolve-next" affordance.
