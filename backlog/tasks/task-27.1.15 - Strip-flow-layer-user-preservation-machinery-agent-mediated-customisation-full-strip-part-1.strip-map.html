Task 27.1.15 · drift · simplification · shipped + reviewed

Stripping the flow-layer preservation machinery

Customisation is agent-mediated: the agent regenerates and re-applies descriptions and chart edits. No diagram layer holds direct human byte-edits that an agentic pass must avoid clobbering. The recovery apparatus built across task 27.1 to protect those edits therefore guards a state that cannot occur. This is the case for deleting it, the exact removed-vs-kept boundary, and the two seams that change shape.

part 1 of 2 · flow layer + recovery surface package @code-charter/drift + one core writer 8 acceptance criteria · no shims
The premise being acted on

At the flow layer, node descriptions are agent-generated at flow creation; the layer has no user-customisable fields. Customisation at the higher flow-chart layer is agent-mediated too — so no layer holds direct human byte-edits an agentic pass must not overwrite.

1 · The argument

Why this is safe to delete now

Every removed piece exists to answer one question: "a human edited this diagram by hand — how do we carry that edit across a code change without clobbering it?" The machinery below traces that single intent through the flow layer.

The core claim The remap, the bin, and the user-tier writes are a chain whose only payload is a hand-authored label or pin. When the agent authors and re-applies that label, there is nothing to carry — the chain has no payload, so it is pure surplus. Removing it cannot lose a human edit, because no human edit lands on a flow layer.

4a · Trickiest seam

Flow identity on a renamed entrypoint

The remap's sole job was to carry a user label across a flow-id change when the dominant seed got renamed. Without it, the question is: does the existing seed-gone soft-delete cover that case? It does — the renamed entrypoint simply re-hydrates as a fresh flow.

One reviewer-surfaced subtlety: for the seed-gone code flow to reach the soft-delete, it has to be surfaced as a re-sync candidate in the first place. affected_persisted_flows in affected_flows.ts now returns flow.member_edges.length === 0 for a zero-seed flow — code flows induce their members from the seeds, so they carry no enumerated member edges (skill/doc flows do). That single predicate is what routes the superseded code flow into resync_persisted_flow, where it is retired.

4b · Trickiest seam

write_descriptions on a soft-deleted side-node

Previously, when an agentic.description side-node sat soft-deleted (binned), write_descriptions skipped it — "leave it for drift.resolve." With drift.resolve's reattach arm and the bin both gone, that skip would strand an agentic description forever. The shipped fix is simpler than a conditional restore: every side-node is upserted unconditionally with deleted_at: null, which resurrects a soft-deleted node and overwrites any prior content in one step — no existing lookup, no separate store.restore call.

for (const item of ordered) {
  const id = description_node_id(item.symbol_path);
  const existing = existing_by_id.get(id);
  if (existing?.deleted_at != null) {
    skipped.push(item.symbol_path); // in the re-attachment bin — leave it for drift.resolve
    continue;
  }
  // Re-upsert only when agentic-owned; never demote a user-promoted node.
  if (!existing || existing.layer !== "user") {
    store.upsert_node({ id, kind: DESCRIPTION_NODE_KIND, /* …anchor onto live symbol… */ });
  }
  // Always (re)write the side-node live at the agentic tier: deleted_at: null resurrects a
  // soft-deleted node and overwrites any prior content, so a stranded description is regenerated.
  store.upsert_node({
    id, kind: DESCRIPTION_NODE_KIND, path: item.file_path,
    anchor: format_anchor({ symbol_path: item.symbol_path, content_hash: item.content_hash }),
    layer: "agentic", attributes: {}, field_ownership: {}, origin: "describe-policy",
    intent_source: "code-edit", deleted_at: null,
  });
  store.write_fields({ kind: "node", id }, { description: item.text, description_hash: item.content_hash, description_source: item.source }, "agentic");
  written.push(item.symbol_path);
}
Why both branches collapse The existing.layer !== 'user' guard protected a user-promoted description node from demotion, and the deleted_at != null skip parked binned nodes for drift.resolve. At this layer descriptions are agentic-only and there is no bin to park into, so neither branch has a job — both go. The loop body is now a single unconditional upsert_node({ … deleted_at: null }) with no existing lookup and no store.restore call. The skipped field is gone too: WriteDescriptionsResult is now just { written }.
