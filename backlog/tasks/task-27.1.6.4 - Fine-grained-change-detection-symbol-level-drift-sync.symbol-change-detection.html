task-27.1.6.4 · drift / change-detection

Drift follows the code change, not the file save

The reconcile engine reduces a turn's edits to a structured, symbol-level change set, then scopes re-sync and re-describe to exactly the symbols that moved. An edit to a shared file that touches no symbol a flow depends on no longer re-syncs that flow.

What changes, and why it matters

Today the unit of change is the file. The Stop hook hands reconcile(file_set) the files edited this turn; a persisted flow re-syncs whenever any member's defining file intersects that set — regardless of whether the symbol that actually changed is in the flow, and regardless of whether the edit changed any symbol's body at all. Editing one helper in a shared file re-syncs every flow that reaches anything else in that file, and a comment-only edit still triggers a full re-sync and re-render.

The symbol-level signal already exists: re_extract classifies every anchored node as hit / body-changed / relocated / miss against its symbol_path:content_hash anchor — but that verdict drives description preservation only. Nothing aggregates those verdicts into a "which symbols changed this turn" delta, and nothing uses it to scope which flows re-sync. This task computes and uses that delta.

The two payoffs

Correctness — no spurious whole-flow re-syncs off an unrelated edit in a shared file. Cost — the reconcile stops doing downstream work (re-syncs, re-describes, flow writes) the delta proves unnecessary.

The reconcile pipeline: before & after

The chain is unchanged from the substrate up: re_extract still refreshes raw, still emits its findings (existing consumers untouched). What changes is one extra return value from re_extract and the predicate inside affected_persisted_flows.

Re-induce the flow's members; re-sync if a changed symbol is a member (body drift) or the membership set itself changed (structural drift).

The SymbolDelta — reusing verdicts that already exist

The delta is a pure function in core: diff the fresh resolver index of the changed files against the store's persisted anchors (the symbol_path:content_hash carried by anchored description/flow nodes). Each class is one of resolve_anchor's verdicts, lifted from per-node to per-turn. Ariadne supplies the structure; the resolver's existing content_hash supplies body identity.

Delta classResolver verdictMeaningDetected by
added (no anchor) A fresh symbol_path with no persisted anchor. Fresh index entries not present in the persisted anchor set.
modified body-changed Body changed under a stable symbol_path (content_hash differs, path matches). resolve_anchor arm 2.
relocated relocated symbol_path changed under a stable body (rename-in-place or cross-file move). resolve_anchor arm 3 (content matches at a different path).
removed miss Neither path nor body resolves (a simultaneous rename + re-body). resolve_anchor arm 4.

Caveat to keep in view

added is "relative to anchored knowledge", not "new to the repo". An existing-but-never-described code symbol reads as added the first time its file is reconciled. This is harmless: downstream work is scoped through flows, so an added symbol only does anything if it lands in a flow's induced membership.

It rides on what re_extract already does

reconcile_node already computes the verdict per anchored node. The change is additive: re_extract returns the aggregate SymbolDelta alongside its existing findings. Every current .findings consumer is untouched.

The id-space join (the correctness crux)

Three id spaces describe the same callable, and the re-sync predicate must cross all three. This is the subtle part: the delta is keyed in resolver space, but flow membership is computed in call-graph space.

The bridge

anchored_symbols_from_ariadne emits, per callable in one pass, the symbol_id, the symbol_path, and the content_hash together — the only place all three meet. build_symbol_path_index(graph) gives the symbol_path → SymbolId inverse. A modified symbol_path is mapped through these to its live SymbolId before testing flow membership.

Note the third space is deliberately different: flow_id_of drops the enclosing chain (entrypoints are top-level, so enclosing is []). The membership-change test (b) compares the flow's fresh induced member set as flow_id_of values against the stored anchor_set — so be careful not to compare an enclosing-aware symbol_path against an enclosing-free flow_id_of value. They are mapped through the live graph, not string-compared across spaces.

The (a) + (b) re-sync trigger — why both

Per CODE flow (skill/doc flows with no code seeds are skipped — handled elsewhere), two complementary symbol-level triggers fire a re-sync. They catch orthogonal kinds of drift, so both are required.

  • (a) body change. A modified symbol — mapped to its live SymbolId via the bridge — is in the flow's induced membership. The flow's contents changed even though its shape did not.
  • (b) membership change. The flow's fresh induced member set (as flow_id_of values) differs from its stored anchor_set. This catches added / removed / relocated members — the flow's shape changed.

Why (a) alone is insufficient: a new call edge can pull a previously-unrelated symbol into the flow without that symbol being "modified" relative to its own anchor. Why (b) alone is insufficient: editing the body of a symbol already in the flow changes no membership but absolutely needs a re-sync. Together: re-sync iff body OR membership drifted.

packages/drift/src/reconcile/affected_flows.ts — the predicate that changes (AC#2)
return persisted.filter((flow) => {
  if (is_skill_or_doc_flow(flow)) return false;  // no code seeds → handled elsewhere
  const membership = reconstruct_flow_membership(rows_of(flow), graph);
  const members = induce_members(membership, graph);

  for (const member of members) {                                  // TODAY: changed-FILE intersection
    const file = adapter.file_of(member) ?? node_path(member);
    if (file !== undefined && changed_files.has(file)) return true;
  }
  return false;
  // (a) a modified SYMBOL (mapped to its live SymbolId) is a member
  for (const sp of delta.modified) {
    const id = symbol_path_index.get(sp);
    if (id !== undefined && members.has(id)) return true;
  }
  // (b) the induced membership set (as flow_id_of) differs from the stored anchor_set
  const fresh = new Set([...members].map((id) => flow_id_of(graph.nodes.get(id)!)));
  return !set_equal(fresh, flow.anchor_set);
});

No-op on non-semantic edits (falls out for free)

AC#4 is not separate work — it is a consequence. content_hash trims and normalizes EOL but does include in-body comments, so a comment edit is not literally invisible at hash level. But a comment-only or whitespace-only edit that changes no symbol body produces a delta that is empty for that flow: neither (a) nor (b) fires, so there is no flow write, no last_synced_at churn, and no description rewrite.

One subtlety worth flagging in review

content_hash strips the symbol's own identifier (so a rename stays hash-stable) and includes in-body comments. A comment change therefore does shift content_hash — it would surface as modified. The no-op guarantee holds at the flow level: a comment edit to a symbol no flow depends on reconciles nothing. Confirm this matches the intended AC#4 boundary before building the comment-only fixture.

Cost posture — what the delta saves, what it doesn't (AC#5)

The whole-repo Ariadne re-index per Stop-hook run remains. The bin is a fresh process each turn, and persisting Ariadne's incremental Project across processes needs a daemon or serialization Ariadne does not expose — larger than this task.
