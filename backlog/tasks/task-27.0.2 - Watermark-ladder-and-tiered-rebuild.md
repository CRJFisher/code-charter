## Description

<!-- SECTION:DESCRIPTION:BEGIN -->

Preservation across regeneration — the load-bearing guarantee of the whole model. Implements the field-granularity watermark on the precedence ladder `user (2) > agentic (1) > raw (0)` and the per-tier rebuild, so a cheap, deterministic raw re-parse never clobbers an expensive agentic field, and an explicit agentic pass never clobbers an irreplaceable user edit.

The ladder is what lets the disposable raw tier be nuked and rebuilt at will while the preserved tiers above it survive untouched.

<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria

- [x] #1 `write_fields(target, fields, as_tier)` writes a field only if `TIER_RANK[current_owner] <= TIER_RANK[as_tier]`, stamps every written field as `as_tier` in `field_ownership`, and returns `{ skipped: string[] }` for the fields a higher tier owns; a field absent from `field_ownership` is treated as raw-owned (rank 0 per `TIER_RANK`)

- [x] #6 Tests cover the full ladder matrix (raw/agentic/user × raw-pass/agentic-pass/user-edit) and the rebuild-preserves-higher-tiers property on a `:memory:` store

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->

Builds on `@code-charter/core` (task-27.0.1): `SqliteGraphStore` already provides `write_fields` (the field-ownership ladder) and a minimal `rebuild_layer` that nukes a layer's rows and runs the writer in one transaction. This task adds the watermark-driven invocation and the per-layer disposable-cache clearing (a `TODO(task-27.0.2)` in `sqlite_graph_store.ts` marks the spot).

1. Implement `write_fields` as the single ladder-respecting write: per field, compare `TIER_RANK[current_owner]` against `TIER_RANK[as_tier]` (`json_extract`/`json_set` on `field_ownership` is the SQLite encoding of that engine-agnostic comparison); write the fields that pass, collect the rest into `skipped`.
2. Route every producer through it — raw re-parse writes `as_tier='raw'`, the agentic pass `'agentic'`, user edits `'user'`. The agentic pass stamps a generated `description` agentic-owned so a later user edit can promote it.
3. Implement `rebuild_layer(layer, write)`: in a transaction, delete the layer's rows plus any cache `table_disposition()` marks disposable (e.g. `anchor_resolution`), run `write`; higher tiers and their watermarked fields survive because the writer goes through `write_fields`.
4. Tests: ladder matrix, dual-sourced description promotion (raw-absent → agentic → user), rebuild-preserves-higher-tiers, soft-deleted survival, `table_disposition` consulted (spy).

## Implementation Notes

<!-- SECTION:NOTES:BEGIN -->

### Narrative summary

A Code Charter diagram is a peer artifact above the code: it carries the *what* and *why*, holds
agent- and human-authored meaning, and must **survive code edits** rather than be regenerated on
demand. That survival is the whole reason the graph is split into three tiers by cost of
regeneration — `raw` (free, deterministic parse output, the only disposable tier), `agentic`
(expensive LLM output, preserved), and `user` (irreplaceable, preserved). This task implements the
mechanism that makes "survive" true in practice: a cheap raw re-parse must never wipe out an
expensive agentic description, and an agentic re-run must never wipe out a hand-authored edit.

The guarantee rides on **two independent axes**, kept deliberately separate so each stays simple:

- **The per-field ladder** (`write_fields`) decides, field by field, whether a write may land: a
  write at tier T overwrites a field only if the field's current owner ranks at or below T, then
  restamps it as T-owned (an unstamped field counts as raw-owned). This is the single
  ladder-respecting write every producer goes through, so a raw pass can refresh raw fields without
  ever touching an agentic or user field on the same row.
- **The per-tier rebuild** (`rebuild_layer`) is the disposable-tier reset: it nukes only the named
  tier's live rows plus every cache the registry marks disposable, then runs the producer's writer
  inside one transaction. A raw re-parse is `rebuild_layer('raw')`; an explicit, run-when-asked
  agentic pass is `rebuild_layer('agentic')`; the user tier is never rebuilt.

The point of keeping the axes apart is that `rebuild_layer` needs **no knowledge of field ownership**.
Higher tiers survive a rebuild structurally — their rows carry a different `layer` and so are never in
the delete set — and any higher-tier-owned field on a row the writer *does* rewrite is protected by
the ladder the writer already goes through. That is why the disposable raw tier can be torn down and
rebuilt at will while the preserved tiers above it sit untouched: the precedence ladder
`user (2) > agentic (1) > raw (0)` is the load-bearing invariant, and the rebuild leans on it rather
than re-deriving it.

### What changed

`write_fields` already existed from task-27.0.1; the work here was completing `rebuild_layer`
(`packages/core/src/storage/sqlite_graph_store.ts`), which previously stubbed the rebuild and left a
`TODO(task-27.0.2)`. It gained two behaviors beyond that stub:

1. **Preserve soft-deleted higher-tier rows (AC#5).** The delete is scoped `WHERE layer = ? AND
   deleted_at IS NULL`. A soft-deleted agentic/user row keeps its `deleted_at` flag through the
   rebuild — neither hard-deleted nor un-flagged — so it stays restorable. (Raw rows can never be
   soft-deleted, so the clause is a no-op for `rebuild_layer('raw')`.)
2. **Clear disposable caches as data (AC#3, AC#4).** After deleting the tier's rows, the rebuild
   iterates `table_disposition()` and `DELETE`s the rows of every table tagged `disposable` (today
   just `anchor_resolution`). It reads the registry as data — no hard-coded name list — so a future
   disposable cache is cleared with zero code change. `DELETE`, not `DROP`: the cache table survives;
   only its stale derived rows go.

### Preservation boundary (open, deferred to task-27.1 / task-27.2)

`write_fields` deliberately does **not** touch the structural `layer` column — it scopes to the
overflow attribute bag only. So a row's `layer` and its per-field ownership are independent, and a
user-promoted field can come to sit on a `layer='agentic'` (or `'raw'`) row. Because `rebuild_layer`
deletes by **layer**, such a row is **hard-deleted** by a rebuild of that layer whenever the writer
does not re-emit its id — and the ladder cannot protect a row that no longer exists. This is a real
data-loss path for irreplaceable content and is in tension with task-27.0's "agentic/user content is
never hard-deleted."

Closing it is a model-level decision (it touches the task-27.0.1 `write_fields` contract and
task-27.0.4 render-layer semantics), so it is **not resolved in this task**. The field-level ladder is
closed; the row-level guarantee (a higher-tier-owned field never strands on a rebuild-eligible layer)
is an invariant the producers must uphold. The candidate mechanisms — promote the row's `layer` on
edit, require the rebuild writer to re-emit kept rows, or add an ownership-aware guard in
`rebuild_layer` — are recorded in the matching notes on **task-27.1** (which runs the destructive
rebuild) and **task-27.2** (which authors user-owned fields) for a decision there.

### Tests

`sqlite_graph_store.test.ts` gained a `task-27.0.2` block (61 tests pass overall):

- **Full ladder matrix** — `{absent, raw, agentic, user}` owner × `{raw, agentic, user}` pass,
  parametrized; each cell asserts overwrite-and-restamp vs. skip-and-leave-untouched (AC#1/#2/#6).
- **Dual-sourced `description`** — raw-absent → agentic default (stamped agentic-owned) → user
  promotion, after which both a raw re-parse and an agentic pass are skipped (AC#2).
- **Rebuild preserves higher tiers** — `rebuild('raw')` keeps agentic+user rows (and cascades raw
  edge provenance away); `rebuild('agentic')` keeps user rows; and a surviving row's higher-tier-owned
  fields are protected when the rebuild writer rewrites them through `write_fields`, for both node and
  edge targets (AC#3).
- **Disposable-cache clear is data-driven (AC#3, AC#4)** — a spy asserts `table_disposition()` is
  consulted on every rebuild, and a second assertion proves the *returned data* drives the `DELETE`
  (mocking the disposition flips which table is cleared). A file-backed test seeds two disposable
  caches (one registered after seeding) plus a registered-preserved cache and asserts both disposable
  ones are emptied while the preserved one and the cache tables themselves survive; another asserts
  the cache clear rolls back atomically when the writer throws.
- **Soft-deleted survival** — a soft-deleted agentic row keeps its flag through `rebuild('agentic')`
  and restores afterward (AC#5).
