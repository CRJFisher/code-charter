The crux — two tiers, one shared seam

27.1.6.6 made the drift-sync skill orchestrate the agent over three thin bin modes. The bin is now deterministic and dumb; the judgement is prose in SKILL.md and the sub-agent. So the harness splits in two: Tier 1 pins the deterministic plumbing by feeding golden JSON through the same three modes — fast, no model, runs in CI. Tier 2 measures the only part that needs a model — does the real agent stitch what is real and decline what is not — by driving the actual skill via claude -p, then scoring the store. Both tiers exercise the identical three modes production uses; only the judgement slot differs: golden JSON, or the live agent.

The spine — the same three modes, two ways to fill the judgement slots

A single vertical chain runs through every fixture: list → judgement-1 → apply-stitch → judgement-2 → apply-descriptions. Tier 1 fills both judgement slots with committed golden JSON and asserts the store. Tier 2 fills them with the live drift-reconciler agent and scores the store against authored ground truth.

The three fixtures — two must stitch, one must decline

Each fixture is a subdirectory of real TypeScript (≤100 lines) under packages/drift/src/reconcile/__fixtures__/stitch_eval/, parsed by real Ariadne. The pattern dictates the ground truth: a dispatch/registry or callback indirection Ariadne cannot resolve must stitch; two independent domains must not.

1registry_dispatch/
expected · STITCH 1 umbrella · seeds ≥ 2

dispatch calls handlers[key]() — a registry lookup Ariadne cannot resolve — so each registered handler is its own orphan entrypoint. The agent must stitch the dispatcher + handlers into one umbrella.

2callback_wiring/
expected · STITCH 1 umbrella · seeds ≥ 2

Both callers pass a function into schedule(run), which invokes run() — unresolved. The agent must recognise the callback contract and stitch the callers with the scheduler.

3unrelated_pair/
expected · DECLINE 2 singleton flows

Two independent domains with no shared unresolved call site. The agent must decline to stitch — the false-positive guard that proves it does not merge whatever sits nearby.

Tier 1 — golden JSON through the three modes, in CI

reconcile_stitch_eval.test.ts spawns the built drift-reconcile bin over each fixture — exactly the boundary 27.1.6.6's reconcile_stitch.test.ts already proves on its own dispatch fixture. No model; golden wire JSON stands in for the agent's judgement. It guards the deterministic contract the live agent depends on.

golden JSON enters exactly where the agent's would

The bin takes the judgement only as a JSON file on disk — stitch.json for the umbrellas, descriptions.json for the member text. Tier 1 supplies that file the same way the live agent does: write the golden JSON, shell the bin, read the store. So the contract Tier 1 pins is exactly the one Tier 2's agent must satisfy — only the author of the JSON differs.

Tier 2 — drive the real agent, score the store, tune the prompt

A runnable stitch_eval script — gated behind STITCH_EVAL_LIVE=1 + an API key, excluded from default CI — runs the actual drift-sync skill / drift-reconciler sub-agent per fixture via claude -p. The agent itself runs the three modes; the harness only sets the stage, then inspects and scores. This is the point of the task: the feedback loop a prompt edit re-runs.

the loop, per fixture
  • 1Scaffold a throwaway repo + graph store from the fixture; stage its files as the pending-reconcile set the skill fetches.
  • 2Run the real flow via claude -p — the drift-reconciler sub-agent invokes drift-sync and follows list → judge-stitch → apply-stitch → judge-describe → apply-descriptions.
  • 3The agent drives the three modes itself — runs --list-entrypoints, Reads/Greps the source, writes stitch.json + descriptions.json, calls --apply-stitch / --apply-descriptions.
  • 4Inspect the resulting store — read back flows, bridges, and member descriptions.
  • 5Score against authored ground truth and emit a per-fixture report; edit SKILL.md / the agent prompt and re-run in one command to re-score.
what each tier is for
  • T1Guards the structural / replay contract. Deterministic, no model, runs in CI on every push — catches a regression in the bin or the JSON contract the moment it lands.
  • T2Measures the live semantic judgement. Does the agent stitch registry_dispatch / callback_wiring and decline unrelated_pair? Token-costing, key-gated — run when tuning the prompt.
  • ·The score is a number you generate. 2 stitches + 1 decline + all members described = pass; the report shows the agent's chosen umbrellas and rationale so a miss is diagnosable, not a bare red X.
  • ·claude -p over the Agents SDK — whichever most faithfully reproduces the production sub-agent run; the choice is documented inline in the harness.
deferred

Captured-batch fixtures from a real production reconcile wait until at least one live run has been observed; the mini-codebase harness is the iteration loop until then.
