The crux — two tiers, one shared seam

27.1.6.6 made the drift-sync skill orchestrate the agent over three thin bin modes. The bin is deterministic and dumb; the judgement is prose in SKILL.md and the sub-agent. So the harness splits in two over the same three modes. Tier 1 pins the deterministic plumbing by feeding golden JSON through the modes — fast, no model, in CI. Tier 2 measures the only part that needs a model — does the real agent stitch what is real and decline what is not — by driving the actual skill via claude -p against four committed mini-codebases, then scoring the store. Only the judgement slot differs: golden JSON, or the live agent. The first live run scored 4/4 PASS at haiku.

The spine — the same three modes, two ways to fill the judgement slots

A single vertical chain runs through every fixture: list → judgement-1 → apply-stitch → judgement-2 → apply-descriptions. Tier 1 fills both judgement slots with committed golden JSON and asserts the store. Tier 2 fills them with the live drift-reconciler agent — driven by the Stop hook's verbatim instruction over the installed .claude bundle — and scores the store against authored ground truth.

The four fixtures — three must stitch, one must decline

Each fixture is a subdirectory of real source (≤100 lines) under packages/drift/src/reconcile/__fixtures__/stitch_eval/, named by the Ariadne resolution weakness it contains, parsed by real Ariadne. Every shape is empirically pinned by running --list-entrypoints over the candidate before any golden JSON is written. The pattern dictates the ground truth: a dispatch/registry, callback, or untyped-receiver indirection Ariadne cannot resolve must stitch; two independent domains must not.

1dynamic_key_dispatch/TS
expected · STITCH 1 umbrella · seeds 3

dispatch runs return fn(); where fn came from lookup_handler(key) over a Map registry — a lookup Ariadne cannot resolve. Registration is deliberately out of band (in-code registration would let Ariadne track the reference, and nothing would fragment), so each handler is its own orphan entrypoint. The agent must stitch the dispatcher + both handlers into one umbrella.

2untyped_callback_invocation/TS
expected · STITCH 1 umbrella · seeds 2

boot and shutdown each pass a named callback into run_scheduled(run), which invokes run() — unresolved, so each caller orphans. Both trees carry the shared scheduler.ts:11 run(); site as evidence. The agent must recognise the callback contract and stitch the callers with the scheduler.

3untyped_receiver_method/Python
expected · STITCH 1 umbrella · seeds 2

run_item calls item.process() on a parameter with no type annotation, so Ariadne emits no call node at all and Item.process orphans. The corroborable bridge site is therefore the run_item(Item()) constructor line in main's tree — the unresolved-site evidence the inventory offers, repeated once per parsed call node on that line. The agent must stitch main, run_item, and the method into one umbrella.

4control_unrelated_pair/TS
expected · DECLINE 2 singleton flows

Two independent domains with no shared unresolved call site — neither carries any unresolved_sites. The agent must decline to stitch — the false-positive guard that proves it does not merge whatever sits nearby.

resolved call unresolved / missing call site (?) the hidden edge the agent recovers orphan entrypoint = stitch seed reachable member, not a seed

Tier 1 — golden JSON through the three modes, in CI

reconcile_stitch_eval.test.ts spawns the built dist/bin/drift_reconcile.js over a tmp copy of each fixture — per-process Ariadne isolation, the full CLI contract for free — and replays golden wire JSON through the three modes. 13 tests, in the package's serial chain. No model. It guards the deterministic contract the live agent depends on.

golden JSON enters exactly where the agent's would

The bin takes the judgement only as a JSON file on disk — stitch.json for the umbrellas, descriptions.json for the member text. Tier 1 supplies that file the same way the live agent does: write the golden JSON, shell the bin, read the store. So the contract Tier 1 pins is exactly the one Tier 2's agent must satisfy — only the author of the JSON differs.

The two-id-space join — what the Python fixture surfaced

Building untyped_receiver_method exposed a real product seam in apply_descriptions: the agent submits a description under the flow-layer symbol_path the stitch returned (processor.py#process:method), but the description row must persist under the anchor's enclosing-qualified, rename-stable path (processor.py#Item.process:method). Those two id-spaces only line up by joining through the graph index on symbol_id — the same join hydration's describe step uses. Without it, a stitched method's description silently skipped.

sharp edge — left for a follow-up

Flow-layer symbol_paths drop the enclosing scope, so two same-named methods in one file collapse to one path and the first-sorted symbol wins the index — a latent, pre-existing ambiguity none of the four fixtures triggers, deferred.

Tier 2 — drive the real agent, score the store, tune the prompt

src/bin/stitch_eval.ts — gated behind STITCH_EVAL_LIVE=1, run as npm run stitch_eval [fixture], excluded from default CI — runs the actual drift-reconciler sub-agent per fixture via claude -p at haiku by default. The agent itself runs the three modes; the harness only sets the stage, then inspects and scores. This is the point of the task: the feedback loop a prompt edit re-runs. The first live run scored 4/4 PASS.

the loop, per fixture
  • 1Scaffold a throwaway repo from the fixture; install_drift copies the production .claude bundle from assets/ (an assets/ prose edit reaches the next run with no copying); stage the fixture files as the pending-reconcile set.
  • 2Run the real agent via claude -p with build_reconcile_instruction() — the Stop hook's verbatim instruction — pointing CODE_CHARTER_DB at the throwaway store. STITCH_EVAL_MODEL overrides the haiku default; STITCH_EVAL_KEEP=1 keeps the temp repos.
  • 3The agent drives the three modes itself — runs --list-entrypoints, Reads/Greps the source, writes stitch.json + descriptions.json, calls --apply-stitch / --apply-descriptions.
  • 4Inspect the resulting store — read back live flows, bridges, and member descriptions.
  • 5Score against authored ground truth and write a per-fixture report to packages/drift/.stitch_eval_runs/; edit SKILL.md / drift-reconciler.md in assets/ and re-run in one command to re-score.
what each tier is for
  • T1Guards the structural / replay contract. Deterministic, no model, runs in CI — catches a regression in the bin or the JSON contract the moment it lands. 13 tests over the four fixtures.
  • T2Measures the live semantic judgement. Does the agent stitch the three weakness fixtures and decline control_unrelated_pair? Token-costing, key-gated — run when tuning the prompt. 4/4 PASS at haiku on the first run.
  • ·The score is a number you generate. 3 stitches + 1 decline + every member llm-described = pass; the report carries the agent's chosen umbrellas and rationale so a miss is diagnosable, not a bare red X.
  • ·claude -p over the Agents SDK — print mode loads the exact installed .claude prose under measurement; the SDK would re-encode it and diverge from the artifact being tuned (and is not a repo dependency). Documented inline.
excluded from CI by construction

The live tier is an npm script, never a jest suite — so it never runs in CI, where it would spend real tokens and need an authenticated claude CLI. Tier 1 guards the deterministic contract this live run rides on.
